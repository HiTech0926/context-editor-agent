# Codex 风格消息流与上下文地图方案

更新时间：2026-04-22

## 目标

这一版不是单纯改样式，而是把消息流、工具调用展示、思考态展示和上下文地图的分层关系理顺。

目标有 4 个：

1. 主聊天区的 `正在思考...` 看起来更接近 Codex
2. 工具调用不是统一堆在消息末尾，而是按生成顺序插入正文
3. 工具展示不要再是厚重卡片，而是更轻的正文内折叠行
4. 上下文地图忠实于“产品自己的原始上下文”，而不是直接平铺 OpenAI 的 wire format

---

## 一、思考态要改成什么样

### 目标效果

- 只显示一行淡一点的 `正在思考...`
- 没有边框
- 没有背景块
- 不做胶囊
- 文本本身有横向流光扫过
- 如果当前链路没有安全可展示的 reasoning summary，就不做假展开

### 设计原则

这个状态应该更像“流式生成中的状态行”，不是“独立提示组件”。

---

## 二、工具调用为什么不能继续挂在消息末尾

当前实现里，一条 assistant 消息基本还是：

- `text`
- `toolEvents`

这种结构会导致所有工具事件天然只能挂在最后。

但期望效果是：

1. 一段正文
2. 一个工具调用块
3. 再一段正文
4. 再一个工具调用块
5. 最后收尾正文

所以 assistant 消息要从“单文本 + 末尾工具数组”，升级成“按顺序排列的内容块”。

---

## 三、消息结构分层

### 1. Provider Payload

这是发给 OpenAI Responses API 的最终编译结果。

它应该允许这些 item 混排：

- `message`
- `function_call`
- `function_call_output`

这一层是协议层，不直接作为用户看到的聊天记录。

### 2. Context IR

这是项目内部真正应该稳定保存的中间表示。

建议 assistant turn 内部按块保存：

- `text_block`
- `tool_call_block`
- `tool_output_block`
- 以后可扩展 `reasoning_summary_block`

这层是“原始上下文”的产品定义版本。

### 3. UI Projection

这是给用户看的最终界面。

主聊天区展示 assistant turn 时，应按 Context IR 的顺序渲染块，而不是直接拿 Provider Payload 平铺。

---

## 四、上下文地图怎么组织

### 顶层

上下文地图默认按回合展示：

- 用户一轮
- 助手一轮
- 用户一轮
- 助手一轮

不要默认把 assistant 的内部步骤全部炸开。

### assistant 展开后

展开某一轮 assistant 时，再看到内部顺序：

- 文本块
- 工具调用
- 工具结果
- 文本块
- 工具调用
- 工具结果

这样既保留了“原始执行顺序”，又不会让地图顶层爆炸。

### 为什么不能直接平铺协议层

如果直接把 OpenAI 的 wire format 平铺到地图里，一轮复杂任务可能会变成：

- user
- assistant
- function_call
- function_call_output
- assistant
- function_call
- function_call_output
- assistant

这样虽然协议上真实，但用户心智会崩，而且地图会迅速膨胀。

所以地图应忠实于 `turn -> blocks` 的结构，而不是忠实于 flat protocol items。

---

## 五、工具调用在主聊天区怎么展示

### 目标效果

主聊天区里的工具块应尽量接近 Codex：

- 无外层卡片
- 无厚重边框
- 先是一行轻量灰字：`调用了 1 个工具`
- 再是一行工具标题，可折叠
- 展开后才看结果

### 不该展示什么

- 默认不要展示机器参数 JSON
- 默认不要展示一整坨 schema
- 默认不要把工具 name + arguments + result 全部堆出来

### 应该展示什么

- 工具名称
- 工具做了什么
- 结果摘要
- 需要时再展开看详情

---

## 六、长工具结果的处理

长工具结果不应该把整条消息撑爆。

建议做法：

1. 默认先显示摘要
2. 展开后出现固定高度的结果区块
3. 这个区块内部可滚动
4. 外层消息高度保持稳定
5. 以后可加“查看完整结果”或单独详情面板

这比直接在聊天流里塞超长原文稳定得多。

---

## 七、这轮实现优先级

### 第一优先级

- 把 assistant 消息改造成顺序块结构
- 思考态改成无框流光文本
- 工具块插回正文流，不再统一挂在尾部

### 第二优先级

- 工具折叠行视觉向 Codex 靠拢
- 工具结果区改成轻量固定高 + 内部滚动

### 第三优先级

- 上下文地图改成顶层 turn、展开看 assistant blocks

---

## 八、当前不做的事

- 不直接显示 raw chain-of-thought
- 不把 provider wire format 当成上下文地图默认视图
- 不为了“像 Codex”把底层协议和产品存储结构混在一起

---

## 九、一句话总结

这次要做的不是“把工具卡片改扁一点”，而是：

**把 assistant 回复从一整块文本，改成一串按生成顺序组织的内容块。**

只有这样，主聊天区、工具展示、思考态和上下文地图才会同时变顺。
