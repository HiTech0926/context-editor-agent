## 1 技术架构概览

```
┌─────────────────────────────────────────────────────────┐
│                   Electron Shell                        │
├───────────────┬─────────────────────────────────────────┤
│  React 前端    │  Python 后端 (web_server.py)             │
│               │                                         │
│  App.tsx      │  AppState          ← 全局状态管理         │
│  ContextMap   │  SessionState      ← 每个会话的状态       │
│  Sidebar.tsx  │  ContextWorkbench  ← 上下文模型回合管理    │
│  Workbench    │  Draft             ← 工作快照             │
│  SettingsView │  Adapters          ← 多 provider 适配     │
│               │                                         │
│  Vite + TS    │  ThreadingHTTPServer + SimpleAgent       │

```

- **前端**：React + TypeScript + Vite，入口 `react_app/src/App.tsx`

- **模型适配**：`agent_runtime/adapters/` 下有 OpenAI、Claude、Gemini 等适配器
- **持久化**：本地 JSON 文件 `data/hash_web_state.json`

## 2 数据结构

### 2.1 SessionState

每个会话在后端的核心数据结构（`web_server.py` ~L109）：

```python
@dataclass(slots=True)
class SessionState:
    session_id: str
    title: str
    scope: str                                    # "chat" | "project"
    project_id: str | None
    agent: SimpleAgent                            # 主聊天 agent 实例
    transcript: list[dict[str, object]]           # 唯一数据源
    context_input: list[dict[str, object]]        # provider 格式的上下文输入
    context_workbench_history: list[dict[str, str]]  # 手动页对话历史
    context_revisions: list[dict[str, object]]    # 版本历史
    pending_context_restore: dict[str, object] | None  # 待撤回的恢复
    active_request_mode: str | None               # "main" | "context" | None
    active_request_id: str | None                 # 当前请求 ID（并发锁）
    active_cancel_event: threading.Event | None   # 取消事件
```

### 2.2 TranscriptRecord

后端 Python 定义（`agent_runtime/core/transcript_contract.py`）：

```python
@dataclass(slots=True)
class TranscriptRecord:
    role: TranscriptRole              # "user" | "assistant"
    text: str                         # 消息正文
    attachments: list[AttachmentRecord]
    blocks: list[TranscriptBlock]     # 展示用内容块
    tool_events: list[ToolEventRecord]
    canonical_items: list[CanonicalItem]  # 协议层结构
    provider_raw: ProviderRaw | None
    metadata: JsonObject
```

前端 TypeScript 定义（`react_app/src/types.ts`）：

```typescript
interface TranscriptRecord {
  role: ContextInputRole;
  text: string;
  attachments?: AttachmentRecord[];
  blocks?: MessageBlock[];
  toolEvents?: ToolEvent[];
  providerItems?: ProviderItem[];  // 前端仍用此名，后端为 canonical_items
}
```

> **命名差异说明**：后端统一使用 `canonical_items`，前端 types.ts 里两者并存（`providerItems` 和 `canonical_items`），运行时等价。

### 2.3 CanonicalItem

协议层核心结构（`agent_runtime/core/canonical_types.py`）：

```python
@dataclass(slots=True)
class CanonicalItem:
    type: CanonicalItemType    # "message" | "tool_call" | "tool_result"
    role: TranscriptRole | None
    content: JsonValue
    name: str                  # 工具名
    call_id: str
    arguments: JsonValue
    output: JsonValue
    status: CanonicalStatus | str
    provider_raw: ProviderRaw | None
    metadata: JsonObject
```

### 2.4 一轮对话 = 一个 Record

一个 assistant record 内部的 `canonical_items` 可以包含任意组合：

```
message(assistant) → tool_call → tool_result → message(assistant) → tool_call → ...
```

**只要属于同一轮助手回复，都挂在同一条 record 下面。** 前端按 `blocks` 渲染展示，后端按 `canonical_items` 构建模型请求。

### 2.5 Token 估算

token 数**不持久化在 record 上**，由前端通过 `countTokens()` 实时估算（`react_app/src/utils.ts`）。上下文模型对话时，概览信息（token 数、工具名、调用次数、preview）由后端 `overview_items()` 动态生成。

## 3 上下文地图实现

### 3.1 组件结构

- `ContextMapSidebar.tsx`（~1060 行）：上下文地图主组件
- `ContextMapSidebar.polish.css`：样式文件
- 内部管理：节点展开状态、选中状态、minimap、虚拟滚动

### 3.2 节点编号逻辑

```typescript
// 只有 user 和 assistant (an) 角色的节点参与编号
const isEditable = isEditableContextNode(message.role);  // user | an
const editableNodeNumber = isEditable ? editableNodeCursor + 1 : null;
```

system / developer / compaction / context 等角色的节点不参与编号。

### 3.3 负载颜色

由 `contextTokenWeight.ts` 管理，根据阈值配置计算 `weightClass`：

- `light`（绿色）：token < warningThreshold
- `medium`（黄色）：warningThreshold ≤ token < criticalThreshold
- `heavy`（红色）：token ≥ criticalThreshold

阈值通过设置页配置，存储在 settings 的 `context_token_warning_threshold` 和 `context_token_critical_threshold`。

### 3.4 Minimap

- 每个节点一个横条，高度按 role + weightClass 决定（4~7px）
- Viewport 矩形根据主列表滚动比例定位
- 支持拖动 viewport 矩形来控制主列表滚动
- 通过 `minimapDragRef` 管理拖动状态

### 3.5 节点选中

- 状态管理：`selectedIndexes: Set<number>`
- 单击：toggle 单个节点
- Ctrl/Cmd 点击：additive toggle
- 拖选：通过 `selectionDragRef` 管理拖动状态，`buildRangeSelection()` 计算范围
- 拖选自动滚动：`ensureSelectionAutoScroll()` + `requestAnimationFrame` 循环
- 虚拟化：节点数 > 80 且无展开节点时启用虚拟滚动

## 4 Working Draft 实现

### 4.1 ContextWorkbenchDraft

位于 `web_server_modules/context_workbench.py`，是一轮上下文模型对话的工作对象：

```python
class ContextWorkbenchDraft:
    def __init__(self, transcript, selected_indexes):
        self.nodes = [...]           # 从 transcript 构建节点列表
        self.selected_node_numbers = [...]  # 选中节点编号
        self.operations = []         # 本轮操作记录
        self._working_version = 0    # 本轮草稿版本号，每次 mutation 递增
```

**不是完整的 transcript 深拷贝**，而是基于标记系统：每个节点有 `active` 标记，操作时改标记而非复制数据。

### 4.2 节点结构

```python
@dataclass
class ContextWorkbenchDraftNode:
    source_record: dict              # 原始 transcript record
    source_node_number: int          # 快照编号
    active: bool                     # 是否仍然有效
    items: list[...]                 # item 列表（用于详情展开）
```

### 4.3 生命周期

```
用户在手动页发送消息
  ↓
创建 ContextWorkbenchDraft（基于当前 transcript + 选中索引）
  ↓
`run_context_chat_turn()` + `ContextWorkbenchToolRegistry` 管理本轮对话和工具调用
  ↓
模型先拿到自动注入的初始快照（全局概览）
  ↓
模型按需展开详情 / 执行操作
  ↓  (操作改 draft 的节点标记和 items)
  ↓  (mutation 工具只返回本次变化 delta)
  ↓  (左侧上下文地图不更新)
  ↓  (节点编号冻结)
  ↓
所有计划内编辑完成后，模型调用 confirm_working_snapshot
  ↓  (返回最终 active 节点概览)
  ↓
模型调用 set_context_revision_summary 写版本说明
  ↓
模型结束输出
  ↓
如果有 mutation:
  → apply_context_workbench_mutation()
  → 新 transcript 写入 session
  → 生成 revision
  → 左侧上下文地图刷新 + 重新编号
如果无 mutation:
  → 不生成 revision
```

## 5 工具系统实现

### 5.1 工具定义

每个工具是一个 `ContextWorkbenchToolDefinition`（~L136）：

```python
@dataclass(slots=True)
class ContextWorkbenchToolDefinition:
    name: str
    label: str
    description: str
    parameters: dict[str, Any]      # JSON Schema
    status: str                     # "available" | "preview"
    handler: Callable[[dict], ToolExecution]
```

### 5.2 工具清单

| 工具名 | 构建方法 | 行号 |
|---|---|---|
| `get_context_node_details` | `_build_node_detail_tool()` | `context_workbench.py` |
| `delete_context_item` | `_build_delete_item_tool()` | `context_workbench.py` |
| `replace_context_item` | `_build_replace_item_tool()` | `context_workbench.py` |
| `compress_context_item` | `_build_compress_item_tool()` | `context_workbench.py` |
| `compress_context_nodes` | `_build_compress_nodes_tool()` | `context_workbench.py` |
| `delete_context_nodes` | `_build_delete_nodes_tool()` | `context_workbench.py` |
| `confirm_working_snapshot` | `_build_confirm_working_snapshot_tool()` | `context_workbench.py` |
| `set_context_revision_summary` | `_build_set_revision_summary_tool()` | `context_workbench.py` |

> `preview_context_selection` 已移除。初始节点概览在每轮开始时自动注入给模型，不再通过工具重复读取。

### 5.3 目标定位实现

目标定位遵循一个边界：**模型负责判断，工具只执行显式参数**。

`resolve_target_nodes()` 只接受模型传入的明确节点编号：

```python
def resolve_target_nodes(self, arguments):
    # 1. 使用显式 node_numbers
    explicit = normalize_node_numbers(arguments.get("node_numbers"), ...)
    if explicit:
        return self._nodes_by_number(explicit)

    # 2. 兼容旧参数 node_indexes，但不做语义推断
    legacy = normalize_node_numbers(arguments.get("node_indexes"), ...)
    if legacy:
        return self._nodes_by_number(legacy)

    return []
```

`selected_node_numbers` 只出现在初始快照和少量错误返回里，作用是给模型参考，不参与工具自动定位。工具层不支持自然语言目标提示参数、本地智能定位、按当前选中节点兜底，也不会在缺少明确参数时擅自修改全部 active 节点。缺少 `node_numbers` 时，工具返回 `target_resolution`，由模型重新根据当前快照或详情明确参数后再调用。

### 5.4 节点详情缓存

`ContextWorkbenchRound` 维护 `_returned_detail_node_numbers: set[int]`，记录已返回详情的节点。再次请求同一节点时会提示"已缓存"，避免重复传输。

### 5.5 工具返回信息原则

上下文工作区采用“开头全量、中间增量、最后确认”的信息流：

1. **每轮开始**：`build_context_workspace_snapshot()` 自动注入当前上下文快照。user 节点给全文，assistant 节点给概览。
2. **查看详情**：只有需要 assistant 节点内部 item 级内容时，才调用 `get_context_node_details`。
3. **执行修改**：`delete / replace / compress` 类 mutation 工具只返回本次变化 `mutation_delta`，不再重复返回全部节点概览。
4. **最终确认**：所有计划内编辑完成后，调用一次 `confirm_working_snapshot`，返回最终所有 active 节点概览，以及 deleted / compressed 等 inactive 节点去向。
5. **版本说明**：`set_context_revision_summary` 只保存恢复页摘要，不再返回完整概览。

`mutation_delta` 主要包含：

```json
{
  "payload_kind": "mutation_delta",
  "summary": "Compress nodes #3-4",
  "change_type": "compress",
  "working_version": 2,
  "changed_nodes": [3, 4],
  "active_node_count": 8,
  "inactive_node_count": 2,
  "changed_node_details": [...]
}
```

`confirm_working_snapshot` 返回最终概览：

```json
{
  "payload_kind": "final_working_snapshot",
  "working_version": 2,
  "active_node_count": 8,
  "inactive_node_count": 2,
  "active_nodes": [...],
  "inactive_nodes": [
    { "node_number": 3, "status": "compressed", "replaced_by": "Draft Node A" }
  ],
  "operations": [...]
}
```

## 6 Revision 系统实现

### 6.1 Revision 数据结构

```python
{
    "id": "uuid",
    "revision_number": 3,
    "label": "Revision 3",
    "summary": "压缩了 Node #4 的工具输出",
    "created_at": "2026-05-09T12:00:00Z",
    "change_type": "compress",
    "change_types": ["compress"],
    "changed_nodes": [4],
    "snapshot": [...],                              # 完整 transcript 快照
    "context_workbench_history_snapshot": [...],     # 手动页历史快照
    "operations": [...],                            # 操作详情
    "operation_count": 1,
    "node_count": 8,
    "is_active": True,
}
```

### 6.2 关键函数

| 函数 | 说明 |
|---|---|
| `ensure_initial_context_revision()` | 确保存在初始 revision（revision_number=0） |
| `sync_active_context_revision_snapshot()` | 同步当前 active revision 的 snapshot |
| `build_context_revision_entry()` | 构建新的 revision 条目 |
| `mark_active_context_revision()` | 设置某个 revision 为 active |
| `find_active_context_revision_id()` | 查找当前 active 的 revision ID |
| `context_revision_summaries()` | 生成发给前端的摘要列表（不含 snapshot） |

### 6.3 恢复与撤回

**恢复**（`restore_context_revision()`）：
1. 找到目标 revision
2. 保存当前状态到 `pending_context_restore`（用于撤回）
3. 把 transcript 切换到目标 snapshot
4. 手动页历史也回退
5. 标记目标 revision 为 active

**撤回**（`undo_context_restore()`）：
1. 从 `pending_context_restore` 恢复之前的 transcript 和手动页历史
2. 清空 `pending_context_restore`
3. 标记原来的 active revision

**撤回失效**：任何新操作（主聊天发消息、上下文模型对话）都会把 `pending_context_restore` 清空。

## 7 并发控制实现

通过 `acquire_session_request()` / `release_session_request()` 实现请求级互斥锁：

```python
def acquire_session_request(self, session, mode):
    # mode = "main" | "context"
    if active_mode and active_mode != mode:
        raise ValueError("当前主聊天和上下文工作区不能并行")
    if active_mode == mode:
        raise ValueError("当前这条主对话还没结束")
    # 分配 request_id + cancel_event
```

- 主聊天使用 `mode="main"`
- 上下文模型使用 `mode="context"`
- 两者互斥，同 mode 也互斥
- 支持通过 `cancel_session_request()` 取消正在进行的请求

## 8 前后端通信

### 8.1 上下文模型对话

- 接口：`POST /context_chat`（流式）
- 请求：`{ session_id, message, selected_node_indexes, reasoning_effort }`
- 流式事件：`delta` / `reset` / `reasoning_start` / `reasoning_done` / `tool_event` / `error` / `done`
- `done` 事件包含：更新后的 history、conversation、context_input、revisions、pending_restore

### 8.2 恢复操作

- 恢复：`POST /restore_context_revision` → `{ session_id, revision_id }`
- 撤回：`POST /undo_context_restore` → `{ session_id }`

### 8.3 Bootstrap

- `GET /init` 返回全量初始数据，包含所有 session 的 conversations、context_inputs、context_workbench_histories、context_revision_histories、pending_context_restores

### 9.2 分层功能解析

整个项目分为 **5 个功能层**，从外到内：

#### 层 1：Electron 桌面壳层 — `electron/`

**职责**：把 Web 应用包装成桌面应用。

- `main.js` 是 Electron 主进程入口，做三件事：
  1. **启动 Python 后端**：自动找到可用端口，用 `child_process.spawn` 拉起 `web_server.py`
  2. **创建浏览器窗口**：加载 `http://127.0.0.1:{port}/react`，无边框 + 自定义标题栏
  3. **桥接原生能力**：通过 IPC 提供窗口控制（最小化/最大化/关闭）、文件夹选择对话框等
- `preload.js` 把 `electronAPI`（minimize/maximize/close/selectFolder 等方法）安全地暴露给渲染进程

#### 层 2：React 前端 — `react_app/`

**职责**：用户界面和交互逻辑。

| 模块 | 关键文件 | 功能 |
|---|---|---|
| 主框架 | `App.tsx` | 全局状态管理（sessions/projects/settings）、路由调度、流式消息处理 |
| 聊天区 | `ChatView.tsx` + `Composer.tsx` + `MessageList/Bubble/Content.tsx` | 消息展示 + 输入 + Markdown/代码渲染 |
| 上下文地图 | `ContextMapSidebar.tsx` | 左侧节点列表 + minimap + 多选拖选 + 虚拟滚动 |
| 上下文工作区 | `ContextWorkbench.tsx` | 右侧手动对话面板 + revision 列表 + 恢复/撤回操作 |
| 侧边栏 | `Sidebar.tsx` | 项目/会话列表 + 搜索 + 新建/删除/归档 |
| 设置 | `SettingsView.tsx` + `SettingsProvidersPanel.tsx` | 多 Provider 管理 + API Key 配置 + 偏好设置 |
| 模型 | `ChatModelPicker.tsx` + `modelGroups/Capabilities/Logos.ts` | 模型选择 UI + 能力展示 + 分组 |
| 基础设施 | `api.ts` + `types.ts` + `utils.ts` + `i18n.ts` + `contextTokenWeight.ts` | API 封装 / 类型 / 工具函数 / 国际化 / Token 权重 |

#### 层 3：Python 后端 — `web_server.py` + `web_server_modules/`

**职责**：HTTP 服务、会话状态管理、上下文工作区编排。

- `web_server.py`（~3000 行）是后端总调度：
  - `AppState` 管理全局状态（所有会话、项目、设置）
  - `SessionState` 管理单个会话（transcript、context_input、revisions 等）
  - HTTP Handler 注册所有 API 路由（`/api/chat`、`/api/context_chat`、`/api/init` 等）
  - 并发控制（`acquire/release_session_request`）
  - 持久化到 `data/hash_web_state.json`

- `web_server_modules/` 是从主服务拆出的功能模块：

  | 模块 | 功能 |
  |---|---|
  | `context_workbench.py` | 上下文工作区所有逻辑：Draft 节点系统、8 个工具的定义和 handler、Revision 构建/恢复、snapshot 生成 |
  | `transcript.py` | Transcript 处理：provider 格式转换、流式 think-tag 解析、blocks 构建、text 提取 |
  | `attachments.py` | 附件上传/持久化/Data URL 解析 |
  | `providers.py` | Provider 模型列表获取、类型判断、URL 构建 |
  | `paths.py` | 统一路径常量（STATE_DIR、ATTACHMENTS_DIR、REACT_DIST_DIR 等） |
  | `serialization.py` | JSON 安全序列化（处理 surrogate pair 等边界情况） |

#### 层 4：SimpleAgent — `simple_agent/`

**职责**：高层 Agent 封装，管理一个完整的对话+工具调用循环。

| 文件 | 功能 |
|---|---|
| `agent.py` | SimpleAgent 类：持有对话历史、调用 adapter 发请求、执行工具循环、支持流式输出 |
| `config.py` | Settings dataclass + JSON 配置文件的加载/保存/合并 |
| `tools.py` | 内置工具定义（read_file / write_file / run_command 等） |
| `codex_tool_registry.py` | Codex 风格工具注册表（动态注册/注销工具） |
| `provider_clients.py` | 根据 provider 类型构造 HTTP 客户端（设置 base_url + API Key） |
| `desktop_bridge.py` | Electron 桌面环境桥接辅助（如检测是否在 Electron 内运行） |

#### 层 5：Agent Runtime — `agent_runtime/`

**职责**：最底层的模型 API 适配抽象，让上层代码不关心具体用的是哪家模型。

- `core/`：定义了所有"协议层"的数据类型和事件
  - `canonical_types.py`：`CanonicalItem`（统一的消息/工具调用/工具结果表示）
  - `stream_events.py`：流式事件类型（TextDelta / ToolCallReady / Error / ProviderDone 等）
  - `agent_core.py`：Agent 运行时核心循环（prompt → stream → tool_call → tool_result → 继续）
  - `transcript_contract.py`：TranscriptRecord 数据契约

- `adapters/`：每个 AI provider 一个适配器，都继承 `BaseAdapter`
  - `chat_completions_adapter.py`：OpenAI Chat Completions 格式
  - `responses_adapter.py`：OpenAI Responses 格式
  - `claude_adapter.py`：Anthropic Claude 格式
  - `gemini_adapter.py`：Google Gemini 格式

### 9.3 各层如何串联

下面是一个完整请求的生命周期，展示各层如何协作：

```
用户操作（点击发送）
  │
  ▼
┌─ 层 2：React 前端 ─────────────────────────────────────┐
│  App.tsx 发起 POST /api/chat（流式）                     │
│  api.ts 构造 fetch 请求                                  │
│  EventSource 接收 SSE 事件                               │
│  ChatView / MessageContent 实时渲染                      │
└────────────────────┬───────────────────────────────────┘
                     │ HTTP（SSE 流式）
                     ▼
┌─ 层 3：Python 后端 ────────────────────────────────────┐
│  web_server.py Handler 接收请求                         │
│  AppState.acquire_session_request() 获取互斥锁           │
│  session.agent.run_turn_stream() 开始模型对话             │
│  Handler 将 stream 事件转为 SSE 推送给前端                │
│  AppState.append_turn() 持久化对话记录                    │
│  AppState.release_session_request() 释放锁               │
└────────────────────┬───────────────────────────────────┘
                     │ 方法调用
                     ▼
┌─ 层 4：SimpleAgent ───────────────────────────────────┐
│  agent.py 构建 input_items（历史 + 系统提示 + 用户消息）  │
│  选择合适的 adapter                                     │
│  调用 adapter.stream() 发起模型请求                      │
│  如果模型返回 tool_call → 执行工具 → 再次调用模型         │
│  循环直到模型输出纯文本回复                                │
└────────────────────┬───────────────────────────────────┘
                     │ 方法调用
                     ▼
┌─ 层 5：Agent Runtime ─────────────────────────────────┐
│  adapter（如 claude_adapter.py）将 canonical 格式        │
│  转为 provider 专有的 HTTP 请求格式                       │
│  发送 HTTP 请求到对应 AI 服务（如 api.anthropic.com）     │
│  将 provider 响应流转换回统一的 StreamEvent 序列           │
│  → TextDeltaEvent / ToolCallReadyEvent / ErrorEvent 等  │
└────────────────────────────────────────────────────────┘
```

**上下文工作区的请求走的是类似但不同的路径**：

```
用户在上下文工作区发送指令
  │
  ▼
前端 ContextWorkbench.tsx → POST /api/context_chat（流式）
  │
  ▼
后端 web_server.py → 创建 ContextWorkbenchDraft
  → ContextWorkbenchToolRegistry 注册 8 个上下文工具
  → 自动注入初始快照给模型
  → 模型调用工具（get_node_details / delete / compress 等）
  → 工具 handler 修改 Draft 节点标记
  → 模型调用 confirm_working_snapshot 确认
  → AppState.apply_context_workbench_mutation() 应用变更
  → 前端上下文地图刷新
```

### 9.4 关键串联点总结

| 串联点 | 连接方 | 机制 |
|---|---|---|
| Electron → Python 后端 | `electron/main.js` → `web_server.py` | `child_process.spawn` 子进程 + HTTP 轮询就绪 |
| Electron → React 前端 | `electron/main.js` → React dist | `loadURL('http://127.0.0.1:{port}/react')` |
| 前端 → 后端 API | `api.ts` → `web_server.py` | HTTP REST + SSE 流式 |
| 后端 → SimpleAgent | `web_server.py` → `agent.py` | Python 方法调用（`agent.run_turn_stream()`） |
| SimpleAgent → Adapter | `agent.py` → `adapters/*.py` | Python 方法调用（`adapter.stream()`） |
| Adapter → AI Provider | `adapters/*.py` → 外部 API | HTTP/HTTPS 请求 |
| 后端 → 上下文工具 | `web_server.py` → `context_workbench.py` | Python 方法调用（Draft + ToolRegistry） |
| 状态持久化 | `AppState` → `data/hash_web_state.json` | JSON 文件读写 |
| 设置持久化 | `config.py` → `data/openai_settings.json` | JSON 文件读写 |
| 前端 ↔ Electron 原生 | React → `preload.js` → `main.js` | IPC（`contextBridge` + `ipcRenderer`） |
