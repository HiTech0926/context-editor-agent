## 1 技术架构概览

```
┌─────────────────────────────────────────────────────────┐
│                   Electron Shell                        │
├───────────────┬─────────────────────────────────────────┤
│  React 前端    │  Python 后端 (web_server.py)             │
│               │                                         │
│  App.tsx      │  AppState          ← 全局状态管理         │
│  ContextMap   │  SessionState      ← 每个会话的状态       │
│  Sidebar.tsx  │  ContextWorkbench  ← 上下文模型回合管理    │
│  Workbench    │  Draft             ← 工作快照             │
│  SettingsView │  Adapters          ← 多 provider 适配     │
│               │                                         │
│  Vite + TS    │  ThreadingHTTPServer + SimpleAgent       │
└───────────────┴─────────────────────────────────────────┘
```

- **前端**：React + TypeScript + Vite，入口 `react_app/src/App.tsx`
- **后端**：Python 单文件服务 `web_server.py`（约 6300 行），内嵌 HTTP 服务器
- **模型适配**：`agent_runtime/adapters/` 下有 OpenAI、Claude、Gemini 等适配器
- **持久化**：本地 JSON 文件 `data/hash_web_state.json`

## 2 数据结构

### 2.1 SessionState

每个会话在后端的核心数据结构（`web_server.py` ~L109）：

```python
@dataclass(slots=True)
class SessionState:
    session_id: str
    title: str
    scope: str                                    # "chat" | "project"
    project_id: str | None
    agent: SimpleAgent                            # 主聊天 agent 实例
    transcript: list[dict[str, object]]           # 唯一数据源
    context_input: list[dict[str, object]]        # provider 格式的上下文输入
    context_workbench_history: list[dict[str, str]]  # 手动页对话历史
    context_revisions: list[dict[str, object]]    # 版本历史
    pending_context_restore: dict[str, object] | None  # 待撤回的恢复
    active_request_mode: str | None               # "main" | "context" | None
    active_request_id: str | None                 # 当前请求 ID（并发锁）
    active_cancel_event: threading.Event | None   # 取消事件
```

### 2.2 TranscriptRecord

后端 Python 定义（`agent_runtime/core/transcript_contract.py`）：

```python
@dataclass(slots=True)
class TranscriptRecord:
    role: TranscriptRole              # "user" | "assistant"
    text: str                         # 消息正文
    attachments: list[AttachmentRecord]
    blocks: list[TranscriptBlock]     # 展示用内容块
    tool_events: list[ToolEventRecord]
    canonical_items: list[CanonicalItem]  # 协议层结构
    provider_raw: ProviderRaw | None
    metadata: JsonObject
```

前端 TypeScript 定义（`react_app/src/types.ts`）：

```typescript
interface TranscriptRecord {
  role: ContextInputRole;
  text: string;
  attachments?: AttachmentRecord[];
  blocks?: MessageBlock[];
  toolEvents?: ToolEvent[];
  providerItems?: ProviderItem[];  // 前端仍用此名，后端为 canonical_items
}
```

> **命名差异说明**：后端统一使用 `canonical_items`，前端 types.ts 里两者并存（`providerItems` 和 `canonical_items`），运行时等价。

### 2.3 CanonicalItem

协议层核心结构（`agent_runtime/core/canonical_types.py`）：

```python
@dataclass(slots=True)
class CanonicalItem:
    type: CanonicalItemType    # "message" | "tool_call" | "tool_result"
    role: TranscriptRole | None
    content: JsonValue
    name: str                  # 工具名
    call_id: str
    arguments: JsonValue
    output: JsonValue
    status: CanonicalStatus | str
    provider_raw: ProviderRaw | None
    metadata: JsonObject
```

### 2.4 一轮对话 = 一个 Record

一个 assistant record 内部的 `canonical_items` 可以包含任意组合：

```
message(assistant) → tool_call → tool_result → message(assistant) → tool_call → ...
```

**只要属于同一轮助手回复，都挂在同一条 record 下面。** 前端按 `blocks` 渲染展示，后端按 `canonical_items` 构建模型请求。

### 2.5 Token 估算

token 数**不持久化在 record 上**，由前端通过 `countTokens()` 实时估算（`react_app/src/utils.ts`）。上下文模型对话时，概览信息（token 数、工具名、调用次数、preview）由后端 `overview_items()` 动态生成。

## 3 上下文地图实现

### 3.1 组件结构

- `ContextMapSidebar.tsx`（~1060 行）：上下文地图主组件
- `ContextMapSidebar.polish.css`：样式文件
- 内部管理：节点展开状态、选中状态、minimap、虚拟滚动

### 3.2 节点编号逻辑

```typescript
// 只有 user 和 assistant (an) 角色的节点参与编号
const isEditable = isEditableContextNode(message.role);  // user | an
const editableNodeNumber = isEditable ? editableNodeCursor + 1 : null;
```

system / developer / compaction / context 等角色的节点不参与编号。

### 3.3 负载颜色

由 `contextTokenWeight.ts` 管理，根据阈值配置计算 `weightClass`：

- `light`（绿色）：token < warningThreshold
- `medium`（黄色）：warningThreshold ≤ token < criticalThreshold
- `heavy`（红色）：token ≥ criticalThreshold

阈值通过设置页配置，存储在 settings 的 `context_token_warning_threshold` 和 `context_token_critical_threshold`。

### 3.4 Minimap

- 每个节点一个横条，高度按 role + weightClass 决定（4~7px）
- Viewport 矩形根据主列表滚动比例定位
- 支持拖动 viewport 矩形来控制主列表滚动
- 通过 `minimapDragRef` 管理拖动状态

### 3.5 节点选中

- 状态管理：`selectedIndexes: Set<number>`
- 单击：toggle 单个节点
- Ctrl/Cmd 点击：additive toggle
- 拖选：通过 `selectionDragRef` 管理拖动状态，`buildRangeSelection()` 计算范围
- 拖选自动滚动：`ensureSelectionAutoScroll()` + `requestAnimationFrame` 循环
- 虚拟化：节点数 > 80 且无展开节点时启用虚拟滚动

## 4 Working Draft 实现

### 4.1 ContextWorkbenchDraft

位于 `web_server.py` ~L3100，是一轮上下文模型对话的工作对象：

```python
class ContextWorkbenchDraft:
    def __init__(self, transcript, selected_indexes):
        self.nodes = [...]           # 从 transcript 构建节点列表
        self.selected_node_numbers = [...]  # 选中节点编号
        self.operations = []         # 本轮操作记录
```

**不是完整的 transcript 深拷贝**，而是基于标记系统：每个节点有 `active` 标记，操作时改标记而非复制数据。

### 4.2 节点结构

```python
@dataclass
class ContextWorkbenchDraftNode:
    source_record: dict              # 原始 transcript record
    source_node_number: int          # 快照编号
    active: bool                     # 是否仍然有效
    items: list[...]                 # item 列表（用于详情展开）
```

### 4.3 生命周期

```
用户在手动页发送消息
  ↓
创建 ContextWorkbenchDraft（基于当前 transcript + 选中索引）
  ↓
ContextWorkbenchRound 管理本轮对话
  ↓
模型通过工具读取概览 / 展开详情 / 执行操作
  ↓  (操作改 draft 的节点标记和 items)
  ↓  (左侧上下文地图不更新)
  ↓  (节点编号冻结)
  ↓
模型结束输出
  ↓
如果有 mutation:
  → apply_context_workbench_mutation()
  → 新 transcript 写入 session
  → 生成 revision
  → 左侧上下文地图刷新 + 重新编号
如果无 mutation:
  → 不生成 revision
```

### 5.3 目标定位实现

所有工具通过 `resolve_target_nodes()` 统一定位目标：

```python
def resolve_target_nodes(self, arguments, ...):
    # 1. 优先用显式 node_numbers
    explicit = normalize_node_numbers(arguments.get("node_numbers"), ...)
    # 2. 其次用 target_hint 做模糊匹配
    # 3. 再其次用当前选中节点
    if allow_selected and self.selected_node_numbers:
        return self._nodes_by_number(self.selected_node_numbers)
    # 4. 最后返回所有活跃节点
```

### 5.4 节点详情缓存

`ContextWorkbenchRound` 维护 `_returned_detail_node_numbers: set[int]`，记录已返回详情的节点。再次请求同一节点时会提示"已缓存"，避免重复传输。

## 6 Revision 系统实现

### 6.1 Revision 数据结构

```python
{
    "id": "uuid",
    "revision_number": 3,
    "label": "Revision 3",
    "summary": "压缩了 Node #4 的工具输出",
    "created_at": "2026-05-09T12:00:00Z",
    "change_type": "compress",
    "change_types": ["compress"],
    "changed_nodes": [4],
    "snapshot": [...],                              # 完整 transcript 快照
    "context_workbench_history_snapshot": [...],     # 手动页历史快照
    "operations": [...],                            # 操作详情
    "operation_count": 1,
    "node_count": 8,
    "is_active": True,
}
```

### 6.2 关键函数

| 函数 | 说明 |
|---|---|
| `ensure_initial_context_revision()` | 确保存在初始 revision（revision_number=0） |
| `sync_active_context_revision_snapshot()` | 同步当前 active revision 的 snapshot |
| `build_context_revision_entry()` | 构建新的 revision 条目 |
| `mark_active_context_revision()` | 设置某个 revision 为 active |
| `find_active_context_revision_id()` | 查找当前 active 的 revision ID |
| `context_revision_summaries()` | 生成发给前端的摘要列表（不含 snapshot） |

### 6.3 恢复与撤回

**恢复**（`restore_context_revision()`）：
1. 找到目标 revision
2. 保存当前状态到 `pending_context_restore`（用于撤回）
3. 把 transcript 切换到目标 snapshot
4. 手动页历史也回退
5. 标记目标 revision 为 active

**撤回**（`undo_context_restore()`）：
1. 从 `pending_context_restore` 恢复之前的 transcript 和手动页历史
2. 清空 `pending_context_restore`
3. 标记原来的 active revision

**撤回失效**：任何新操作（主聊天发消息、上下文模型对话）都会把 `pending_context_restore` 清空。

## 7 并发控制实现

通过 `acquire_session_request()` / `release_session_request()` 实现请求级互斥锁：

```python
def acquire_session_request(self, session, mode):
    # mode = "main" | "context"
    if active_mode and active_mode != mode:
        raise ValueError("当前主聊天和上下文工作区不能并行")
    if active_mode == mode:
        raise ValueError("当前这条主对话还没结束")
    # 分配 request_id + cancel_event
```

- 主聊天使用 `mode="main"`
- 上下文模型使用 `mode="context"`
- 两者互斥，同 mode 也互斥
- 支持通过 `cancel_session_request()` 取消正在进行的请求

## 8 前后端通信

### 8.1 上下文模型对话

- 接口：`POST /context_chat`（流式）
- 请求：`{ session_id, message, selected_node_indexes, reasoning_effort }`
- 流式事件：`delta` / `reset` / `reasoning_start` / `reasoning_done` / `tool_event` / `error` / `done`
- `done` 事件包含：更新后的 history、conversation、context_input、revisions、pending_restore

### 8.2 恢复操作

- 恢复：`POST /restore_context_revision` → `{ session_id, revision_id }`
- 撤回：`POST /undo_context_restore` → `{ session_id }`

### 8.3 Bootstrap

- `GET /init` 返回全量初始数据，包含所有 session 的 conversations、context_inputs、context_workbench_histories、context_revision_histories、pending_context_restores
