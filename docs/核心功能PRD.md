## 1 背景与目标

### 1.1 业务背景

用户和 AI 对话时，很多时候对话的方向会很分散，比如当前讨论的话题已经换方向了，之前的讨论和工具输出还会出现在上下文中，这对需要模型保持高度极致注意力的用户不那么友好，虽然可以使用fork分支对话来解决，但是体验会比较麻烦。

### 1.2 产品目标

**定义：把 AI 的上下文当成一份可编辑文档，而不是一串只能追加的聊天记录。**

核心价值四个字：**可见、可编辑、可恢复、有辅助。**

| 目标 | 说明 |
|---|---|
| **可见** | 用户随时能看到主模型实际吃进去的完整上下文结构 |
| **可编辑** | 用户能像编辑代码一样，精确地修改上下文中的任意一段 |
| **有辅助** | 有一个专门的"上下文模型"帮用户理解和优化上下文 |
| **可恢复** | 所有修改都可以回退到历史版本，不怕改坏 |

### 1.3 成功指标

当以下条件全部成立时，可以认为第一版成功：

1. 用户能看到结构化的上下文地图，而不只是聊天记录
2. 上下文模型能通过节点编号和工具名快速定位任意节点
3. 模型能按需展开详情，并对节点内部做精确修改
4. 修改在本轮结束前不影响正式状态，结束后统一提交
5. 用户能在恢复页看到每次提交记录，并回退到任意历史版本

### 1.4 产品心智

本产品对标 VS Code / Cursor 这类 AI 代码编辑器"按当前文件快照 + 行号改代码"的体验：

| 代码编辑器里的概念 | 对应到我们产品 |
|---|---|
| 一份源文件 | 主模型的上下文 |
| 编辑器辅助（AI 补全等） | 上下文模型 |
| 行号 | 节点编号 `#1 #2 #3` |
| 某一行的局部位置 | 节点内部的 item 编号 |
| 一次编辑回合 | 一轮上下文对话 |
| Git 回退 | 恢复到某个历史版本 |

## 2 用户画像与场景

### 2.1 目标用户

AI 重度使用者——对AI编程有深刻认知，对模型注意力有较高要求，希望能灵活控制上下文的人群。典型代表：AI 辅助编程用户、长文写作用户、研究型对话用户。

### 2.2 核心场景

**场景 A：话题转换**

> 用户使用AI进行bug修复，但在之后用户突然和AI进行了高质量的项目讨论，但上下文已被前面bug修复占据大头，继续聊可能触发全文压缩和模型注意力下降，为了保证心流不被破坏，用户可一句话精准压缩前面bug修复有关讨论。

**场景 B：上下文膨胀导致回答质量下降**

> 用户和 AI 聊了 30 轮，中间有大量已经与当前对话无关工具调用的输出（比如代码执行结果、文件内容）。用户打开上下文地图，发现几条 assistant 节点的 token 数特别高（红色标记），展开一看全是冗长的工具输出。用户通过手动页让上下文模型把这些输出压缩成摘要，上下文瘦身 60%。

**场景 C：上下文诊断**

> 某些时候聊着聊着上下文突然满的很快，用户想知道到底何处出了问题，传统情况只能知道上下文窗口剩余，无法精准定位问题点，上下文地图或者上下文模型可以详细地帮用户分析，在必要时可以自行判断是否保留。

## 3 产品结构

### 3.1 三个区域的关系

| 区域 | 角色 |
|---|---|
| **主聊天区** | 和主模型对话，维护唯一数据源（transcript） |
| **上下文地图** | transcript 的可编辑视图，不是另一套数据 |
| **右侧工作区** | 上下文模型的操作台，包含建议/手动/恢复/设置四页 |

### 3.2 上下文模型

上下文模型是**独立于主聊天模型的第二个模型**，由右侧设置页单独配置 provider 和模型。它不参与主聊天，专门负责分析和修改上下文。
