# 核心功能计划

更新日期：2026-04-23

这份文档只基于当前这轮讨论整理，不参考仓库里已有计划稿。它的作用是作为后续实现“上下文地图 + 上下文模型 + 恢复能力”的长期执行基线。

## 1. 核心目标

本产品的核心不是“又一个 AI 聊天框”，而是：

1. 让用户看见主模型实际吃进去的上下文。
2. 让用户像编辑代码一样编辑上下文。
3. 让上下文模型先理解上下文，再对上下文执行可控修改。
4. 让所有修改都可回退到某个历史版本。

一句话定义：

**把主模型的上下文当成一份可编辑文档，而不是一串只能追加的聊天消息。**

## 2. 第一版产品心智

### 2.1 对齐对象

第一版的逻辑，直接参考 VS Code / Cursor 一类 AI 客户端“按当前文件快照和当前行号改代码”的心智。

对应关系如下：

1. 主模型上下文 = 一份正文档。
2. 上下文模型 = 正在编辑这份文档的辅助模型。
3. 节点编号 `#1 #2 #3` = 当前快照下的行号。
4. 节点内部的 item 编号 = 当前节点内部的局部行号。
5. 一轮上下文对话 = 一次工作区编辑回合。
6. 恢复 = 把整份文档回到某个历史版本。

### 2.2 第一版明确原则

1. 上下文模型看到的节点编号永远以当前快照为准，不追踪旧轮次编号。
2. 上下文模型一轮开始后，当前轮的节点编号冻结，不在工具调用过程中动态改号。
3. 上下文模型工具改的是 working snapshot，不直接改主 transcript。
4. 本轮结束后统一 commit，左侧上下文地图再整体刷新和重新编号。
5. 节点选中只是给模型的强提示，不是使用工具的门槛。
6. 恢复先做成线性回退，不做 selective revert。
7. 主聊天和上下文模型不允许同时运行。

## 3. 第一版明确不做

以下能力不进入第一版：

1. 不做“只撤掉第二次改动，但保留第三到第九次改动”的 selective revert。
2. 不做主聊天和上下文模型同时运行时的session 并发控制。
3. 不强制引入全局稳定 `node_id` 作为产品主语义。
4. 不在一轮上下文模型工具调用过程中实时改动左侧节点编号。
5. 不让恢复页变成一个可随意多次横跳的永久分叉树。

## 4. 总体结构

### 4.1 主聊天

主聊天维护 canonical transcript。它是系统里的“正文档”。

### 4.2 上下文地图

上下文地图是 canonical transcript 的可编辑视图，不是单独的一套数据源。

### 4.3 右侧工作区

右侧工作区是上下文模型的操作台，包含四个页面：

1. 建议
2. 手动
3. 设置
4. 恢复

### 4.4 上下文模型

上下文模型是独立于主模型的第二个模型，由右侧设置页单独配置。

## 5. 上下文地图 UI 规则

### 5.1 三栏结构

上下文地图在 fully expanded 状态下采用三栏结构：

1. 左侧 `node gutter`
2. 中间 `context list`
3. 右侧 `minimap`

### 5.2 节点编号

1. 左侧 gutter 显示 `1 2 3 4 ...` 这样的节点号。
2. 节点号代表当前快照下的顶层节点顺序。
3. 节点号不按 assistant 内部 tool block 编号。
4. 节点号在当前回合结束前不变化。

### 5.3 列表展示

1. 默认所有节点都是缩略态。
2. 缩略态不是摘要，而是原文单行 plain text 预览。
3. 用户消息默认展示原文单行，不做 markdown。
4. assistant 消息默认展示单行 preview，不展开工具细节。
5. 点击节点才展开。
6. 展开后使用小号 markdown 渲染。
7. assistant 展开后按 `blocks` 顺序显示文本块和工具块。
8. 工具块仍然是折叠式，点击后再展开 shell / JSON / 输出内容。

### 5.4 最小化视觉密度

1. 节点之间的垂直间距要压紧。
2. 一屏需要能看到尽可能多的节点。
3. 节点展开和选中不能破坏 minimap 独立性。

### 5.5 minimap 规则

1. minimap 是独立概览，不强行匹配主列表 DOM 高度。
2. minimap 自己可以滚动。
3. minimap 节点是固定高度的横向短条。
4. assistant 条放左侧，user 条放右侧。
5. 颜色只对 assistant 生效。
6. user 节点使用中性灰。
7. minimap 有一个可拖动的 viewport 矩形，表示当前主列表看到的区域。
8. 主浏览以 minimap + 节点选择为主，不依赖原生滚动条作为核心交互。

### 5.6 assistant 节点颜色规则

颜色要表达上下文负载，不只是装饰：

1. 绿色：正常
2. 黄色：偏重
3. 红色：很重

第一版负载优先使用 token 数估算。

### 5.7 节点选中规则

1. 节点可以单击选中。
2. 再点一次取消。
3. 支持 `Ctrl / Cmd` 多选。
4. 支持从 gutter 拖动进行连续批量选中。
5. 拖选经过屏幕边缘时自动滚动。
6. 选中态和展开态是两套状态，不互相替代。
7. 连续选中的节点在输入框里要合并为范围，比如 `节点 #4-7`。
8. 离散选中才单独列出，比如 `节点 #2`、`节点 #4-5`、`节点 #9`。

## 6. 主数据与存储

### 6.1 本地持久化

当前主存储采用本地磁盘持久化。

第一版继续沿用本地存储方案，重点不是换数据库，而是完善上下文版本管理。

### 6.2 session 维度需要持久化的内容

每个 session 需要至少持久化以下内容：

1. `transcript`
2. `manual_history`
3. `context_revisions`
4. `active_revision_index`
5. `last_restore_undo_token`

### 6.3 transcript 的角色

`transcript` 是主聊天和上下文地图共享的数据源。它同时承担两层职责：

1. 展示层
2. 协议层

### 6.4 record 基础结构

每条 transcript record 至少包含：

1. `role`
2. `text`
3. `attachments`
4. `blocks`
5. `toolEvents`
6. `providerItems`
7. `tokenCount`
8. `toolSummary`

说明：

1. `blocks` 主要给主聊天区和上下文地图展开态使用。
2. `providerItems` 是真正接近模型请求协议层的结构。
3. `toolSummary` 是为了给上下文模型快速提供概览信息。

## 7. “用户一轮 / 助手一轮”的规则

### 7.1 展示层规则

前端展示上，坚持：

1. 用户一条消息是一条顶层节点。
2. 助手一轮回复是一条顶层节点。
3. 即使这轮助手中间发生了多个工具调用，展示层仍然视为一个 assistant 节点。

### 7.2 协议层规则

assistant 节点内部的 `providerItems` 不要求固定模板，但要求：

1. 它们都属于同一条 assistant record。
2. 它们可能包含多段 assistant 文本。
3. 它们可能包含多个 `function_call`。
4. 它们可能包含多个 `function_call_output`。
5. 文本和工具调用可以交错出现。

例如，一个 assistant record 内部可能是：

1. assistant text
2. function_call
3. function_call_output
4. assistant text
5. function_call
6. function_call_output
7. assistant text

也可能只有：

1. assistant text

或者：

1. function_call
2. function_call_output
3. assistant text

关键不是固定长相，而是：

**只要属于同一轮助手回复过程，就都挂在同一条 assistant record 下面。**

## 8. 上下文模型初始快照设计

### 8.1 初始快照目标

初始快照不是把全部详情塞给上下文模型，而是先给足够好的全局认知，让模型知道：

1. 当前主上下文由哪些节点组成。
2. 哪些 assistant 节点最重。
3. 哪些 assistant 节点用了什么工具。
4. 当前用户选中了哪些节点。
5. 当前用户到底想操作什么。

### 8.2 user record 的提供方式

默认规则：

1. user record 直接给全文。
2. 不做默认概览。

原因：

1. user 文本通常不长。
2. user 问题本身定义了主模型下一步会往哪个方向回复。
3. 上下文模型看见完整 user 语义更有价值。

### 8.3 assistant record 的提供方式

默认规则：

1. assistant record 初始不直接给全文。
2. assistant record 初始只给概览。

assistant 概览必须包含：

1. `Node #N`
2. `role = assistant`
3. `tokenCount`
4. `tool names`
5. `tool count per tool`
6. `preview`

示例：

```text
Node #4 | assistant | 1.3k tokens | weather x2, shell x1
preview: 我先帮你查一下天气，然后总结今天的变化...
```

### 8.4 选中信息

初始快照里要显式告诉模型当前选中了哪些节点。

但选中信息的语义是：

1. 这些是用户当前重点关注节点。
2. 不是模型唯一能操作的节点。

### 8.5 手动页历史

上下文模型初始输入还要包含：

1. 当前 session 对应的 `manual_history`
2. 当前手动输入

`manual_history` 需要跟主 session 绑定并持久化。

## 9. 节点详情工具设计

### 9.1 工具职责

第一类工具是“获取指定节点详情”。

它的作用是让模型先定位，再决定是否修改。

### 9.2 输入

输入应支持：

1. `node_numbers`

示例：

```json
{
  "node_numbers": [4, 5]
}
```

### 9.3 返回内容

返回必须包含：

1. 目标节点完整文本详情
2. 目标节点完整 `blocks`
3. 目标节点完整 `providerItems`
4. 节点内部 item 编号
5. item 类型
6. 工具名
7. 相关 token 信息

### 9.4 item 编号规则

节点内部 item 按当前快照详情生成局部编号：

1. `item #1`
2. `item #2`
3. `item #3`

这些 item 编号只在当前快照和当前回合语义下有效。

示例：

```text
Node #4
item #1 assistant_text
item #2 function_call(weather)
item #3 function_call_output(weather)
item #4 assistant_text
```

## 10. 操作型工具设计

### 10.1 操作型工具总原则

操作型工具都作用于当前回合的 working snapshot。

它们不直接改 canonical transcript。

### 10.2 目标定位优先级

工具目标定位遵循以下优先级：

1. 用户或模型明确提供的 `node_numbers`
2. 当前选中的节点
3. 全局分析后由模型自行定位出的目标

所以：

1. “没选中”不是报错理由。
2. 选中只是提示，不是门槛。

### 10.3 第一版操作型工具

第一版至少规划以下操作型工具：

1. 删除指定 item
2. 替换指定 item
3. 压缩指定 item
4. 压缩多个节点并生成 summary node

### 10.4 删除指定 item

作用：

1. 删除某个节点里的指定协议层 item。

典型目标：

1. 删除某条 `function_call_output`
2. 删除某段 assistant 文本
3. 删除某个不必要的工具调用块

### 10.5 替换指定 item

作用：

1. 用新内容替换原 item 内容。

典型目标：

1. 把冗长的工具输出替换成压缩版
2. 把一段 assistant 文本改写成更短版本

### 10.6 压缩指定 item

作用：

1. 让模型根据原内容生成压缩版。
2. 再用压缩版替换原 item。

第一版实现上可视为“分析 + 替换”的封装工具。

### 10.7 多节点总结压缩

作用：

1. 选择多个节点。
2. 生成一个新的 summary node。
3. 让当前上下文中原节点退场。

第一版默认行为：

1. 生成一个新的 summary node。
2. 原节点从当前 active transcript 中移除。
3. 恢复时通过 revision 回退整轮操作，不在第一版里对单个来源节点做细粒度 restore。

### 10.8 工具调用后必须返回什么

每个操作型工具返回不能只有“成功”。

必须至少返回：

1. 变更摘要
2. 影响到哪些节点
3. 更新后的目标节点概览
4. 如有必要，更新后的目标节点详情
5. 当前 working snapshot 是否仍有未提交改动

## 11. working snapshot 生命周期

### 11.1 回合开始

当用户在右侧手动页发起一轮对话时：

1. 从当前 active transcript 复制出一份 working snapshot。
2. 生成该快照下的节点编号。
3. 把这份快照作为本轮上下文模型唯一依据。

### 11.2 回合进行中

1. 模型读取全局概览。
2. 模型按需要请求指定节点详情。
3. 模型在 working snapshot 上执行删除、替换、压缩、总结等操作。
4. 左侧正式上下文地图此时不更新。
5. 本轮内节点编号保持冻结。

### 11.3 回合结束

当上下文模型本轮停止输出并结束时：

1. 如果没有 mutation，则不产生新 revision。
2. 如果有 mutation，则把 working snapshot 一次性 commit 为新的 active transcript。
3. 左侧上下文地图整体刷新。
4. 所有节点重新编号。
5. 右侧恢复页新增一条 revision 记录。

## 12. revision 与恢复

### 12.1 revision 生成时机

只有在以下情况才生成 revision：

1. 上下文模型一轮结束，并且本轮对 working snapshot 产生了实际改动。

纯分析、不改内容的轮次，不生成 revision。

### 12.2 revision 保存内容

第一版为了实现简单和稳定，revision 直接保存完整快照，不只保存 patch。

每个 revision 至少包含：

1. `revision_number`
2. `created_at`
3. `summary`
4. `changed_nodes`
5. `snapshot`
6. `change_type`

### 12.3 恢复页职责

恢复页展示的是 revision 列表，而不是零散工具调用日志。

每条记录需要让用户看见：

1. 这是第几次上下文提交
2. 改动摘要
3. 影响了哪些节点
4. 是否是删除、压缩、替换、总结
5. 点击后可以恢复到该 revision

### 12.4 第一版恢复逻辑

第一版恢复只做线性回退：

1. 用户点击“恢复到某个 revision”
2. 当前 active transcript 直接切换到该 revision 的 snapshot
3. 左侧上下文地图整体刷新
4. 节点重新编号

### 12.5 恢复后的撤回规则

恢复之后只提供一个临时能力：

**撤回本次恢复**

规则如下：

1. 仅对最近一次恢复有效。
2. 只在恢复后、尚未继续进行任何新操作时有效。
3. 一旦用户继续主聊天或继续上下文模型对话，这个“撤回本次恢复”入口立即失效。

### 12.6 第一版不做的恢复能力

第一版不做：

1. 保留后续版本的 selective revert
2. 对旧版本中的单次变更做局部摘除
3. 在多条时间线上来回跳转

## 13. 右侧四页职责

### 13.1 建议

建议页主要承担：

1. 给出可疑冗长节点
2. 给出可能可压缩区域
3. 给出上下文结构性建议

第一版可先做成轻量版。

### 13.2 手动

手动页是上下文模型的主操作页。

要求：

1. 具备独立对话历史
2. 对话历史跟主 session 绑定并持久化
3. 输入框支持引用当前选中的节点
4. 引用显示风格类似输入框内附件 / 引用 chips
5. assistant 回复支持 markdown
6. assistant 回复不必在 UI 上显式标注“当前使用模型名称”

### 13.3 设置

设置页至少负责：

1. 配置上下文模型
2. 配置与上下文工作区相关的参数

### 13.4 恢复

恢复页负责：

1. 展示 revision 列表
2. 预览每次修改摘要
3. 支持恢复到某个 revision
4. 在适当时机提供“撤回本次恢复”

## 14. 选中与输入框引用规则

### 14.1 选中的产品语义

选中节点的语义是：

1. 告诉模型“这些是我当前最关注的节点”
2. 降低模型全局扫描成本

而不是：

1. 没选中就不能分析
2. 没选中就不能操作

### 14.2 输入框引用展示

选中节点应直接进入手动输入框，表现得像用户正在引用这些节点。

规则：

1. 连续节点合并为范围 chip
2. 离散节点分别展示
3. 用户能直观看到这次提问引用了哪些节点

### 14.3 模型实际接收方式

模型收到的不是仅凭输入框文本猜的节点范围，而是服务端显式传入：

1. 当前选中节点列表
2. 当前快照节点概览

## 15. 与主聊天的关系

### 15.1 不并行

规则明确：

1. 进行上下文模型对话时，主聊天必须停止。
2. 主聊天生成回复期间，不允许上下文模型对同一 session 工作。

### 15.2 主聊天历史与上下文历史

1. 主聊天 transcript 是 canonical source。
2. 右侧 `manual_history` 是对应 session 的辅助对话历史。
3. `manual_history` 不等于主聊天 transcript，也不直接写进主模型历史。

## 16. 第一版数据契约建议

### 16.1 Node Overview

```text
Node #4

tools: weather x2, shell x1
preview: 我先帮你查一下天气，然后再总结今天的变化...
```

### 16.2 Node Detail

```text

arguments: ...

item #3 function_call_output(weather)

```

### 16.3 Mutation Result

```text
change_summary: 已压缩 Node #4 的天气工具输出
changed_nodes: [4]
updated_overview:
- Node #4 | assistant | 420 tokens | weather x2
dirty: true
```

## 17. 第一版实施顺序

### 阶段 1：上下文快照和概览契约

目标：

1. 先把上下文模型“看到什么”彻底规范化。

后端任务：

1. 给每个 assistant record 生成稳定的概览字段：
   - token 数
   - 工具名
   - 工具调用次数
   - preview
2. 构建上下文模型初始快照：
   - user 全文
   - assistant 概览
   - 当前选中节点
   - manual_history
3. 明确 `get_node_detail` 的返回契约。

前端任务：

1. 确保选中节点能够稳定传入手动输入。
2. 手动页显示当前引用节点范围。

验收标准：

1. 模型不展开详情也能判断哪条 assistant 最长。
2. 模型不展开详情也能判断哪些节点用了指定工具。

### 阶段 2：working snapshot 与操作型工具

目标：

1. 让上下文模型能在一轮里对 working snapshot 做暂存修改。

后端任务：

1. 实现 working snapshot 生命周期。
2. 实现以下工具：
   - 获取节点详情
   - 删除指定 item
   - 替换指定 item
   - 压缩指定 item
   - 多节点生成 summary node
3. 保证每次 mutation 返回变更摘要和更新后的节点信息。
4. 确保本轮内节点编号冻结。

前端任务：

1. 本轮进行中不刷新左侧正式节点列表。
2. 本轮结束后统一刷新。

验收标准：

1. 用户要求“压缩第四条里的工具输出”时，模型能先展开详情，再正确修改对应 item。
2. 用户未选中节点时，模型仍能通过编号和概览完成定位。

### 阶段 3：commit 与 revision

目标：

1. 在回合结束时把 working snapshot 提交为新的上下文版本。

后端任务：

1. 为每次有改动的回合生成 revision。
2. 把 revision 保存为完整 snapshot。
3. 维护 active revision 指针。

前端任务：

1. 回合结束后刷新上下文地图。
2. 节点重新编号并保持 UI 一致。

验收标准：

1. 删除一个节点后，下一轮编号自动递补。
2. summary node 在 commit 后正式进入上下文地图。

### 阶段 4：恢复页

目标：

1. 提供稳定的线性回退能力。

后端任务：

1. 提供 revision 列表接口。
2. 提供“恢复到某个 revision”接口。
3. 提供“撤回本次恢复”临时能力。

前端任务：

1. 新增恢复页。
2. 展示 revision 摘要。
3. 允许点击恢复。
4. 在适当时机显示撤回本次恢复入口。

验收标准：

1. 用户能稳定回到任意一个已有 revision。
2. 恢复后继续新操作，旧的“撤回本次恢复”能力会失效。

### 阶段 5：建议页与交互打磨

目标：

1. 把建议页从占位升级为真正基于上下文分析的辅助页。

后端任务：

1. 输出冗长节点候选
2. 输出工具使用密集节点候选
3. 输出适合压缩的候选区段

前端任务：

1. 建议页可直接跳转和高亮对应节点
2. 建议项可以一键带入手动页

验收标准：

1. 建议页能给出真实可操作的上下文建议，而不是静态文案。

## 18. 第一版成功标准

当以下条件全部成立时，可以认为核心功能第一版跑顺了：

1. 用户能在左侧看到结构化上下文地图，而不只是聊天记录。
2. 上下文模型初始拿到的是 user 全文 + assistant 概览，不是一股脑全量详情。
3. 模型能通过 `Node #N` 和工具名快速定位目标节点。
4. 模型能按需展开详情，再对节点内部 item 做修改。
5. 修改在本轮结束前不影响正式节点编号。
6. 本轮结束后统一提交并重新编号。
7. 用户能在恢复页看到每次上下文提交，并恢复到任意一个历史版本。
8. 恢复后只允许短暂撤回本次恢复，不允许无限横跳。

## 19. 最后定版结论

这套系统第一版的最终定义如下：

1. 主模型上下文是一份可编辑文档。
2. 右侧上下文模型每轮基于当前快照工作。
3. 节点编号只以当前快照为准。
4. user 默认给全文，assistant 默认给概览。
5. 选中节点是提示，不是门槛。
6. 工具先支持看详情，再支持做修改。
7. 修改只改 working snapshot，本轮结束统一 commit。
8. 恢复先做线性回退，只保留“撤回本次恢复”的临时后悔药。

后续实现、设计、联调、验收，统一以这份文档为准。
