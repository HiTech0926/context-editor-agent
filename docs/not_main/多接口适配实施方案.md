# 多接口适配实施方案

## 1. 目标

在 **不改变现有产品功能语义** 的前提下，把当前已经基于 OpenAI Responses 跑通的主聊天与上下文工作区，扩展到另外三类接口：

1. OpenAI Chat Completions
2. Anthropic Claude Messages
3. Google Gemini GenerateContent

这里的重点不是“把请求体改一改”，而是把当前系统里 **和供应商强绑定的部分拆出来**，让产品层继续稳定地工作。

---

## 2. 核心原则

这次适配必须守住下面几条，不然功能表面上能跑，后面一定会越来越乱。

### 2.1 transcript 永远是产品层语义，不是供应商协议语义

无论底层 provider 在一轮 agent 中间发了多少次工具调用、多少段中间文本、多少种特殊消息，最终 transcript 里都只保留：

- 一条 `user record`
- 一条 `assistant record`

也就是说：

- Claude 的 `tool_result`
- Gemini 的 `functionResponse`
- Chat Completions 的 `tool` message
- Responses 的 `function_call_output`

这些都 **不单独长成用户或助手节点**，而是被折叠进这一轮 assistant record 内部。

### 2.2 顶层提示词模块与真实对话历史必须分层

系统提示词、开发者提示词、记忆块、摘要块，这些都不是“真实聊天历史”，因此不能和 transcript 节点混在一起。

产品层应该明确分成：

1. Prompt Blocks
2. Transcript Records
3. Tool Events

### 2.3 provider 的差异只允许存在于 adapter 层

不能让：

- Claude 的 `tool_use`
- Gemini 的 `parts`
- Chat Completions 的 `tool_calls`
- Responses 的 `items`

直接渗透到主产品逻辑里。

主产品只应该关心一套内部结构。

---

## 3. 这次适配要解决的真实问题

当前系统已经基于 Responses 跑通了整套 agent 能力，所以表面问题像是“换参数格式”，但实际上至少有下面几层问题：

1. **提示词位置不同**
   - Responses：顶层 `instructions` 或 `input[].developer`
   - Chat Completions：`messages[]`
   - Claude：顶层 `system`
   - Gemini：顶层 `systemInstruction`

2. **工具调用挂载位置不同**
   - Responses：平级 `item`
   - Chat Completions：挂在 `assistant.message.tool_calls`
   - Claude：挂在 `assistant.content[]`
   - Gemini：挂在 `model.parts[]`

3. **工具结果回传位置不同**
   - Responses：`function_call_output item`
   - Chat Completions：`role: tool`
   - Claude：`role: user` + `tool_result`
   - Gemini：`role: user` + `functionResponse`

4. **流式事件模型不同**
   - Responses 是事件流 + output item
   - Chat Completions 是 delta + tool call 增量
   - Claude 是 block 级流
   - Gemini 是 candidate/content/part 风格

5. **上下文工作区现在认的是一套近似 Responses 的 providerItems**
   - 这套结构未来要能同时承接另外三家

所以这不是一个“多写三个请求函数”的工作，而是一个 **把 provider runtime 从产品层抽离出来** 的工作。

---

## 4. 建议的总架构

## 4.1 分成三层

### 第一层：Product Core

这一层负责产品本身的稳定语义，不关心供应商。

包含：

- transcript 的增量写入
- user record / assistant record 的定义
- 工具循环的整体流程
- streaming 给前端的统一事件
- 上下文工作区与 revision
- token 统计入口

### 第二层：Canonical Model

这是整个适配最重要的一层。它不是某个供应商的原始请求体，而是产品内部统一格式。

至少要有下面这些概念：

#### PromptBlock

- `kind`: `system | developer | memory | summary`
- `text`
- `editable`
- `source`

#### TranscriptRecord

- `role`: `user | assistant`
- `text`
- `attachments`
- `blocks`
- `toolEvents`
- `canonicalItems`

#### Block

- `kind`: `text | tool`
- `text?`
- `tool_event?`

#### CanonicalItem

- `type`: `message | tool_call | tool_result`
- `role?`
- `name?`
- `call_id?`
- `content / arguments / output`

这一层是未来上下文工作区继续编辑的核心对象。

### 第三层：Provider Adapter

每个供应商只负责做两件事：

1. 把 Canonical Model 编译成该接口的请求体
2. 把该接口的响应流解析回 Canonical Events

---

## 5. Adapter 的职责边界

每个 adapter 必须只做这些事：

### 5.1 build_request

输入：

- prompt blocks
- 历史 transcript
- 当前 turn 临时状态
- 工具定义
- 当前 provider 配置

输出：

- 该 provider 的请求体

### 5.2 stream_response

输入：

- provider 请求体

输出：

- 统一事件流

统一事件流建议固定成下面几种：

- `text_delta`
- `tool_call_started`
- `tool_call_ready`
- `tool_result_attached`
- `round_reset`
- `done`
- `error`

### 5.3 normalize_turn_result

把一整轮 provider 响应收束成产品能落库的结果：

- 最终 answer
- 工具事件列表
- blocks
- canonicalItems

### 5.4 optional: estimate_tokens

不同 provider 的 token 统计可能不完全一样，因此允许 adapter 自己实现 token 估算器。

---

## 6. 三个新接口能不能归为一类

可以，但只能归成“同一个实现家族”，不能归成“完全同一套协议”。

建议分成下面两类：

### A 类：turn-centric provider

包括：

- OpenAI Chat Completions
- Claude Messages
- Gemini GenerateContent

它们的共同点：

- 都有“对话轮次容器”
- 工具调用都挂在 assistant/model 这一轮之内
- 工具结果都是下一轮再回传

它们适合共享：

- turn-based request builder 的骨架
- assistant turn assembler
- 多轮工具循环的公共逻辑

### B 类：item-centric provider

包括：

- OpenAI Responses

它的特点是：

- `message` 不是总容器，只是 item 的一种
- `tool_call` 和 `tool_result` 与 message 平级

它不能直接套用 A 类的请求体拼装，但可以共享：

- 产品层 transcript 语义
- 工具循环总流程
- 最终落库结构

所以最终结构应该是：

- 一个 `AgentCore`
- 一个 `TurnCentricAdapterBase`
- 一个 `ResponsesAdapter`
- 三个 `TurnCentricAdapter` 子类

---

## 7. transcript 应该怎么统一

这一部分是整个方案能不能落地的关键。

## 7.1 user record 的统一规则

无论底层 provider 为了工具循环又发送了多少条协议消息，`user record` 只记录：

- 用户真实输入
- 用户真实附件

不记录：

- tool result 回传消息
- provider 内部协议消息
- 应用注入的摘要/记忆

## 7.2 assistant record 的统一规则

assistant record 记录这一整轮 agent 的产品化结果：

- 用户可见的最终回答文本
- 这一轮中发生的工具调用
- 工具输出摘要
- 顺序化 blocks
- canonicalItems

其中：

- 中间“好的我先看一下”这种文本
- 工具调用
- 工具返回
- 最终回答

都可以进入 assistant record 的内部结构，但**不能再拆成额外 transcript 节点**。

## 7.3 provider-specific 协议消息的命运

不同 provider 的协议消息都应该只在 adapter 里出现。

产品层不要直接存：

- Claude 原始 `tool_use`
- Gemini 原始 `functionCall`
- Chat Completions 原始 delta 片段

产品层只存归一化后的：

- `tool_call`
- `tool_result`
- `message`

如果后面确实有调试需要，可以额外存一份：

- `provider_raw`

但它只能是附加信息，不能变成主逻辑依赖。

---

## 8. 具体实施阶段

下面是建议的落地顺序。这个顺序不是为了“好看”，而是为了每一步都能独立验证，不把系统一次性砸开。

## 阶段 0：先定义不允许被破坏的契约

目标：

- 先把产品层固定住

输出物：

1. transcript record 契约文档
2. assistant record 内部结构契约
3. workbench 可编辑 item 契约
4. streaming 给前端的统一事件契约

验收标准：

- 任何 provider 接入后，都必须能输出同一套 transcript JSON 结构

---

## 阶段 1：把当前 Responses 实现“抽象化但不改行为”

目标：

- 先把现在的 Responses 跑通逻辑搬进 `AgentCore + ResponsesAdapter`
- 功能完全不变

任务拆解：

1. 新建 `AgentCore`
2. 新建 `BaseAdapter`
3. 把 Responses 的请求体构造搬进 `ResponsesAdapter.build_request`
4. 把 Responses 的流式解析搬进 `ResponsesAdapter.stream_response`
5. 把 tool loop 仍然留在 `AgentCore`
6. 保证落 transcript 的逻辑完全不变

验收标准：

- 前端行为完全不变
- transcript 完全不变
- context workbench 完全不变

如果这一阶段都做不到无回归，后面三家都别上。

---

## 阶段 2：正式定义 Canonical Model

目标：

- 把现在“看起来像 providerItems，实际已经半 canonical”的结构扶正

任务拆解：

1. 明确 `PromptBlock`
2. 明确 `TranscriptRecord`
3. 明确 `CanonicalItem`
4. 明确 `ToolEvent`
5. 明确 `AssistantRoundState`

特别注意：

- 这里不要直接沿用某家的命名误导心智
- 如果 `providerItems` 这个名字已经不合适，应尽早改名

推荐新命名：

- `canonical_items`
- `runtime_items`
- `turn_items`

验收标准：

- workbench 编辑逻辑只依赖 canonical items，不直接依赖 Responses item shape

---

## 阶段 3：做 turn-centric adapter 基类

目标：

- 为 Chat Completions / Claude / Gemini 提供一个共享骨架

这个基类至少要抽出：

1. prompt blocks 注入点
2. transcript -> provider history 的编译入口
3. 当前 turn 临时状态的注入入口
4. 工具定义编译入口
5. 工具结果回传编译入口

但要允许子类覆盖：

1. 顶层系统提示词放置方式
2. 工具调用提取方式
3. 工具结果回传消息格式
4. 流式事件解析方式

---

## 阶段 4：先接 OpenAI Chat Completions

为什么先接它：

- 与现有 OpenAI 体系最近
- 便于先验证 turn-centric 路线
- 调试成本最低

任务拆解：

1. 把 Prompt Blocks 编译到 `messages[]`
2. 把 transcript 编译成 `messages[]`
3. 识别 `assistant.tool_calls`
4. 执行工具
5. 把工具结果编译成 `role: tool`
6. 继续下一轮直到拿到最终文本
7. 最终编译成统一的 assistant record

重点难点：

- Chat Completions 的 streaming tool call 增量组装
- `tool_calls` 参数增量拼接
- 并行工具调用处理

验收标准：

- 基本问答能通
- 一轮一次工具调用能通
- 一轮多次工具调用能通
- transcript 与 Responses 版本结构一致

---

## 阶段 5：接 Claude Messages

任务拆解：

1. 把 Prompt Blocks 合并编译到顶层 `system`
2. transcript 编译到 `messages[]`
3. 识别 assistant `content[]` 中的 `tool_use`
4. 执行工具
5. 用新的 `role: user` message 携带 `tool_result`
6. 继续工具循环
7. 归一化回 assistant record

重点难点：

- Claude 把工具结果放在 `user` 侧，这里非常容易污染产品层 transcript
- block streaming 的解析需要单独做
- `thinking` 若启用，需要决定是忽略、摘要化，还是作为 provider_raw 保存

验收标准：

- transcript 中绝不能多出伪 user 节点
- workbench 中用户节点数与真实用户输入数一致

---

## 阶段 6：接 Gemini GenerateContent

任务拆解：

1. 把 Prompt Blocks 编译到 `systemInstruction`
2. 把 transcript 编译成 `contents[]`
3. 识别 `parts[]` 中的 `functionCall`
4. 执行工具
5. 用新的 `role: user` content 携带 `functionResponse`
6. 继续工具循环
7. 归一化回 assistant record

重点难点：

- `parts[]` 是更细粒度结构，天然会比 Claude 更碎
- Gemini 的 function calling 与 thought signature 可能影响多轮工具调用效果
- 多模态 part 混排时，canonical item 的转换规则要明确

验收标准：

- 工具循环能完整闭环
- transcript 不被 provider 协议污染
- 与 Chat Completions / Responses 在产品表现上保持一致

---

## 阶段 7：上下文工作区适配

这一步不能放在最后随便补，因为 workbench 是你们产品的核心差异点。

任务拆解：

1. workbench 不再依赖“Responses 风格 providerItems”
2. workbench 详情页改读 canonical items
3. reverse compiler 改为：
   - canonical items -> assistant record
4. item 级编辑操作统一基于 canonical items
5. prompt blocks / memory blocks 的展示与保护逻辑接入

重点：

- Claude / Gemini 不允许因为底层协议特殊，就把工具结果错误显示成用户节点
- 不同 provider 进入 workbench 后，编辑体验必须一致

验收标准：

- 同一轮工具调用，在四家里都能得到同样的 item 级编辑体验

---

## 阶段 8：provider 配置与切换

任务拆解：

1. UI 增加 provider 选择
2. 模型与 provider 解耦
3. 每个 provider 拥有自己的：
   - base URL
   - auth key
   - capability flags
4. provider 切换时，主聊天与工作区走不同 adapter

这里要加一个重要能力位表：

- supports_streaming
- supports_parallel_tool_calls
- supports_reasoning_summary
- supports_system_prompt_in_messages
- supports_top_level_system_prompt
- supports_multimodal_user_input

这样后面很多行为就不用硬编码判断。

---

## 阶段 9：完整回归矩阵

至少要覆盖下面这些 case：

1. 纯聊天，无工具
2. 一轮一次工具调用
3. 一轮多次工具调用
4. 工具调用后继续文本回答
5. 工具报错
6. 流式中断
7. 有附件输入
8. 上下文工作区查看节点详情
9. 上下文工作区 item 级删除 / 替换 / 压缩
10. revision 提交与恢复
11. 版本切回后继续聊天
12. 切换 provider 后历史兼容显示

---

## 9. 实施时最容易失败的地方

## 9.1 过早把三家完全揉成一套请求体模板

这会失败，因为三家只是同属 turn-centric 家族，不是同一个协议。

## 9.2 把 provider 协议消息直接写进 transcript

这会导致：

- 用户节点数变多
- assistant 节点语义变乱
- workbench 编辑对象错位

## 9.3 只做“能发请求”，不做“能落统一 transcript”

这会造成：

- 表面聊天能用
- 一到 workbench / revision / restore 全坏

## 9.4 在抽象之前就同时接三家

这会让 bug 来源完全分不清。

正确顺序必须是：

1. Responses 抽象化
2. Chat Completions
3. Claude
4. Gemini

---

## 10. 建议的文件组织

这里只给建议，不要求一比一照抄。

```text
agent_runtime/
  core/
    agent_core.py
    canonical_types.py
    prompt_blocks.py
    transcript_compiler.py
    stream_events.py

  adapters/
    base.py
    responses_adapter.py
    turn_centric_base.py
    chat_completions_adapter.py
    claude_adapter.py
    gemini_adapter.py

  tools/
    tool_executor.py
    tool_event_compiler.py

  workbench/
    canonical_item_compiler.py
    canonical_item_reverse_compiler.py
```

核心要求只有一个：

**provider-specific 代码要有明确边界。**

## 11. 建议的第一批真实交付顺序

如果现在开始做，不要一上来就追求“多 provider 全通”。建议按下面四个交付点推进。

### 交付点 A：Responses 抽象化完成

目标：

- 功能 100% 不变
- 只是把 current implementation 迁移成 `AgentCore + ResponsesAdapter`

### 交付点 B：Chat Completions 接通

目标：

- 主聊天能跑
- transcript 结构不变
- workbench 可读

### 交付点 C：Claude 接通

目标：

- 主聊天能跑
- 工具闭环能跑
- transcript 不污染

### 交付点 D：Gemini 接通

目标：

- 主聊天能跑
- 工具闭环能跑
- workbench 与 revision 继续可用

## 12. 最终判断

这件事是能做的，但前提不是“再写三个 agent 文件”，而是：

1. 先把当前 Responses 版本抽成公共核心
2. 正式确立一套 provider 无关的 canonical model
3. 再给三个 turn-centric provider 分别写 adapter

如果这个顺序不倒，功能是有机会保持不变的。

如果这个顺序倒了，最后大概率会出现下面这种局面：

- 能聊天
- 能调工具
- 但 transcript、workbench、revision、恢复页全开始出现语义错位

这也是这次方案最想避免的事。
