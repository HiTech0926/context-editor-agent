# React 前端架构入门教学：从一个按钮看懂 Hash Code 前端

这份文档不是源码说明书，而是帮你建立一个前端心智模型。

你现在只需要先记住一句话：

> 前端本质上是在做三件事：展示界面、响应用户操作、和后端交换数据。

在这个项目里，对应关系是：

```text
展示界面        React 组件 + CSS
响应用户操作    useState / 事件处理函数
和后端交换数据  api.ts + Python 后端 /api/*
```

---

## 1. 先从最小的网页开始

传统网页通常由三件套组成：

```html
<button>发送</button>
```

```css
button {
  color: white;
  background: black;
}
```

```js
document.querySelector('button').addEventListener('click', () => {
  alert('发送了');
});
```

它们分别负责：

```text
HTML        页面结构，页面上有什么
CSS         样式，页面长什么样
JavaScript 交互，用户点了之后发生什么
```

React 没有改变这个本质。React 只是换了一种更适合大型应用的写法：把 HTML、部分 JS 逻辑组织成一个个“组件”。

---

## 2. 一个最简单的 React 组件

React 里，一个按钮可以这样写：

```tsx
function SendButton() {
  return <button>发送</button>;
}
```

这里的：

```tsx
<button>发送</button>
```

看起来像 HTML，但它其实叫 **JSX**。

JSX 可以理解为：

> 在 JavaScript / TypeScript 里面写类似 HTML 的界面结构。

浏览器不能直接运行 JSX。项目构建时，Vite 会把 JSX 转换成浏览器能理解的 JavaScript。

所以你之前理解的“React 最后会转回三件套”是对的，只是更准确地说：

```text
React JSX + TypeScript + CSS
        ↓ Vite 构建
浏览器能运行的 HTML / CSS / JavaScript
```

在本项目里，React 的启动入口是：

```text
react_app/src/main.tsx
```

它把整个 App 挂载到：

```text
react_app/index.html
```

里的 `root` 节点上。

---

## 3. 组件为什么有用

如果只有一个按钮，React 看起来没必要。

但真实产品不是一个按钮，而是：

```text
左侧项目列表
中间聊天区
输入框
模型选择器
右侧上下文地图
设置页
Toast 提示
```

如果全部写在一个 HTML 文件里，会很难维护。

React 的做法是拆成组件：

```tsx
function App() {
  return (
    <div>
      <Sidebar />
      <ChatView />
      <ContextMapSidebar />
    </div>
  );
}
```

你可以把 React 组件理解成：

> 可复用、可组合的界面积木。

在这个项目里：

```text
App.tsx                  总控台
Sidebar.tsx              左侧项目/会话列表
ChatView.tsx             中间聊天区域
Composer.tsx             输入框区域
MessageList.tsx          消息列表
MessageBubble.tsx        单条消息外壳
MessageContent.tsx       消息正文
ContextMapSidebar.tsx    右侧上下文地图
SettingsView.tsx         设置页
Toast.tsx                提示消息
```

所以你看任何一个 `.tsx` 文件时，可以先问：

> 它在界面上代表哪一块？

这是看 React 项目的第一步。

## 4. props：父组件给子组件的参数

现在让按钮显示不同文字：

```tsx
function Button(props: { label: string }) {
  return <button>{props.label}</button>;
}
```

使用它：

```tsx
<Button label="发送" />
<Button label="停止" />
```

这里的 `label` 就是 **props**。

props 可以理解为：

> 父组件传给子组件的数据。

这和产品里的“配置”很像。同一个按钮组件，传不同文案，就显示不同按钮。

真实项目里的例子：

```tsx
<ChatView
  messages={currentConversation}
  currentModel={currentModel}
  isSending={isSending}
/>
```

意思是：

```text
App.tsx 把当前会话消息传给 ChatView
App.tsx 把当前模型传给 ChatView
App.tsx 把是否正在发送传给 ChatView
```

所以：

```text
App.tsx     管数据
ChatView    拿数据展示
```

这是本项目前端的核心模式之一。

## 5. 事件：子组件把用户动作告诉父组件

按钮光能显示还不够，要能点击：

```tsx
function Button(props: { label: string; onClick: () => void }) {
  return <button onClick={props.onClick}>{props.label}</button>;
}
```

使用它：

```tsx
<Button
  label="发送"
  onClick={() => {
    console.log('用户点击了发送');
  }}
/>
```

这里的 `onClick` 也是 props，不过它传的是一个函数。

你可以理解为：

```text
父组件：我给你一个按钮，同时告诉你点了以后该通知谁
子组件：我只负责渲染按钮，用户点了我就调用 onClick
```

真实项目里的例子：

```tsx
<Composer
  composerValue={composerValue}
  onChange={(event) => setComposerValue(event.target.value)}
  onSend={handleSend}
/>
```

意思是：

```text
Composer 负责输入框 UI
用户输入时，Composer 调用 onChange
用户点发送时，Composer 调用 onSend
真正怎么保存输入、怎么发送，由 App.tsx 决定
```

这就是 React 里常见的“父组件管状态，子组件管展示和触发事件”。

## 6. state：组件自己的记忆

如果一个按钮要记录点了几次，可以这样写：

```tsx
import { useState } from 'react';

function CounterButton() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      点了 {count} 次
    </button>
  );
}
```

这里的 `count` 是状态，`setCount` 是修改状态的方法。

你可以理解为：

```text
state = 页面当前记住的东西
setState = 修改这个东西，并让界面重新刷新
```

在本项目里，`App.tsx` 有很多状态，例如：

```text
currentSessionId        当前会话 ID
conversations           所有会话的消息
composerValue           输入框内容
currentModel            当前选中的模型
settingsDraft           设置页草稿
contextMap              右侧上下文地图开关和宽度
runningSessionIds       哪些会话正在生成
```

为什么 `App.tsx` 很大？

因为它现在记住了太多东西。

这也是我之前说的：当前架构能跑，但长期最好把会话、设置、流式消息、上下文工作区拆出去。

## 7. 一个输入框怎么工作

先看一个极简输入框：

```tsx
function MessageInput() {
  const [text, setText] = useState('');

  return (
    <textarea
      value={text}
      onChange={(event) => setText(event.target.value)}
    />
  );
}
```

这里有一个很重要的模式：

```text
value     输入框显示什么
onChange  用户改了以后，更新 state
```

也就是说，输入框里的文字不是浏览器自己随便管，而是 React state 在管。

本项目里的输入框在：

```text
react_app/src/components/Composer.tsx
```

但输入框内容真正的状态在：

```text
react_app/src/App.tsx
```

也就是：

```text
App.tsx 保存 composerValue
Composer.tsx 展示 composerValue
用户输入后 Composer 通知 App.tsx 修改 composerValue
```

这就是你看前端时最常见的数据流：

```text
父组件 state
    ↓ props
子组件展示
    ↓ 用户操作
回调函数通知父组件
    ↓
父组件更新 state
    ↓
界面刷新
```

## 8. 一次最简单的 API 通讯

前端为什么要请求后端？

因为前端只负责界面，不应该直接处理所有业务。例如：

```text
保存会话
调用模型
读取项目文件
保存设置
管理上下文版本
```

这些更适合后端做。

一个最简单的 API 请求长这样：

```ts
async function loadHello() {
  const response = await fetch('/api/hello');
  const data = await response.json();
  return data;
}
```

它的意思是：

```text
前端向 /api/hello 发请求
等待后端返回
把返回内容当成 JSON 解析
交给界面使用
```

在本项目里，所有这类请求集中放在：

```text
react_app/src/api.ts
```

例如：

```ts
export function fetchInit() {
  return apiFetch('/api/init');
}
```

产品视角：

> `fetchInit` 就是“前端启动时，去后端拿初始化数据”。

它会拿到：

```text
项目列表
会话列表
历史消息
默认模型
设置
上下文版本记录
```

所以第一次打开应用时，大致发生：

```text
React 启动
    ↓
App.tsx 调用 fetchInit
    ↓
api.ts 请求 /api/init
    ↓
Python web_server.py 返回初始化数据
    ↓
App.tsx 把数据放进 state
    ↓
Sidebar / ChatView / ContextMapSidebar 显示出来
```

## 9. api.ts 怎么用：直接 fetch 和封装的区别

先看一个不封装的写法。

假设设置页要保存默认模型，组件里可以直接这样写：

```tsx
async function handleSaveSettings() {
  const response = await fetch('/api/settings', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      default_model: 'gpt-5.4-mini',
      max_tool_rounds: 8,
    }),
  });

  const data = await response.json();

  if (!response.ok) {
    throw new Error(data.error || response.statusText);
  }

  return data;
}
```

这段代码能工作，但有几个问题：

```text
每个地方都要写 headers
每个地方都要 JSON.stringify
每个地方都要 response.json
每个地方都要判断 response.ok
每个地方都要处理 error
返回数据没有明确类型
```

如果只有一个接口，无所谓。可是这个项目有很多接口：

```text
/api/init
/api/settings
/api/projects
/api/sessions
/api/send-message-stream
/api/context-chat-stream
/api/context-restore
```

如果每个组件都直接写 fetch，代码会变成“每个房间都自己接水管”。

所以项目在 `api.ts` 里先封装了一个基础方法：

```ts
export async function apiFetch<T>(path: string, options: RequestInit = {}): Promise<T> {
  const headers = new Headers(options.headers);

  if (!(options.body instanceof FormData) && !headers.has('Content-Type')) {
    headers.set('Content-Type', 'application/json');
  }

  const response = await fetch(path, {
    ...options,
    headers,
  });

  let data: unknown = {};
  try {
    data = await response.json();
  } catch {
    data = {};
  }

  if (!response.ok) {
    throw new Error(extractResponseError(response, data));
  }

  return data as T;
}
```

这段封装做了几件公共工作：

```text
自动补 Content-Type
统一执行 fetch
统一解析 JSON
统一处理后端错误
统一返回 TypeScript 类型
```

有了 `apiFetch`，具体接口就可以写得很短。

比如获取初始化数据：

```ts
export function fetchInit(): Promise<InitPayload> {
  return apiFetch<InitPayload>('/api/init');
}
```

保存设置：

```ts
export function saveSettingsRequest(payload: {
  default_model: string;
  openai_base_url: string;
  max_tool_rounds: number;
}) {
  return apiFetch<SettingsResponse>('/api/settings', {
    method: 'POST',
    body: JSON.stringify(payload),
  });
}
```

你可以把它理解成两层：

```text
apiFetch              通用快递公司，负责怎么送、失败怎么报错
fetchInit/saveSettings 具体订单，说明送到哪个地址、带什么东西
```

组件使用的时候，就不用关心底层 fetch 细节了。

直接调用：

```tsx
async function loadApp() {
  const initData = await fetchInit();
  setProjects(initData.projects || []);
  setChatSessions(initData.chat_sessions || []);
}
```

保存设置时：

```tsx
async function handleSave() {
  const result = await saveSettingsRequest({
    default_model: currentModel,
    openai_base_url: settingsDraft.openai_base_url,
    max_tool_rounds: Number(settingsDraft.max_tool_rounds),
  });

  setOpenAISettings(result.settings);
}
```

所以区别是：

```text
直接 fetch
组件自己关心请求细节、错误处理、JSON 解析、接口路径

api.ts 封装
组件只关心产品动作，比如“保存设置”“创建会话”“发送消息”
```

再具体一点：

```text
不封装：组件在说“我要 POST /api/settings，并设置 header，再解析 JSON”
封装后：组件在说“我要保存设置”
```

这就是抽象的价值。

不是为了显得高级，而是让代码更接近产品动作。

在这个项目里，`api.ts` 就像一张“前端能调用哪些后端能力”的清单：

```text
初始化
fetchInit

设置
fetchSettings
saveSettingsRequest
fetchProviderModelsRequest

项目/会话
createProjectRequest
createSessionRequest
deleteSessionRequest
renameProjectRequest

主聊天
sendMessageRequest
streamMessageRequest
cancelActiveRequest

上下文工作区
streamContextChatRequest
restoreContextRevisionRequest
clearContextWorkbenchHistoryRequest
```

以后你想判断“某个产品动作有没有接后端”，优先看：

```text
react_app/src/api.ts
```

如果 `api.ts` 里已经有对应函数，组件通常直接调用它。

如果 `api.ts` 里没有，而后端也没有对应 `/api/*`，那说明这个能力大概率还没接好。

## 10. 普通请求和流式请求的区别

普通请求像点外卖：

```text
下单
等待
一次性收到结果
```

流式请求像对方边写边发：

```text
发出问题
后端一点点返回文字
前端一点点显示
最后收到 done
```

普通请求示意：

```ts
const result = await sendMessageRequest(payload);
setMessages(result.answer);
```

流式请求示意：

```ts
await streamMessageRequest(payload, (event) => {
  if (event.type === 'delta') {
    // 追加一小段文字
  }

  if (event.type === 'done') {
    // 收尾，保存最终结果
  }
});
```

本项目聊天生成用的是流式请求，因为模型回答通常很长，用户希望看到它逐字生成，而不是等全部完成。

对应文件：

```text
react_app/src/api.ts       streamMessageRequest
react_app/src/App.tsx      handleSend
web_server.py             /api/send-message-stream
```

理解这一点后，你就知道“为什么聊天消息生成逻辑比普通表单提交复杂”。

因为它要处理：

```text
开始生成
收到增量文本
收到推理内容
收到工具调用
用户点击停止
生成完成
生成失败
```

## 11. types.ts：前后端说同一种语言

一次 API 请求不是只传字符串，它会传结构化数据。

例如一条消息可能长这样：

```ts
{
  role: 'user',
  text: '帮我解释这个文件',
  attachments: [],
  blocks: []
}
```

前端需要知道：

```text
role 有哪些取值
text 是不是必有
attachments 长什么样
后端返回 answer 还是 message
```

这些约定放在：

```text
react_app/src/types.ts
```

你可以把 `types.ts` 理解为：

> 前端和后端的数据合同。

典型类型：

```text
ProjectSummary          项目摘要
SessionSummary          会话摘要
MessageRecord           前端用于展示的消息
TranscriptRecord        后端传回来的原始会话记录
OpenAISettings          设置
ResponseProviderSettings 模型服务商配置
ContextRevisionSummary  上下文版本记录
```

当你看到一个组件 props 里写：

```ts
messages: MessageRecord[]
```

意思就是：

> 这个组件需要一组“消息记录”来渲染。

这比单纯看函数名更可靠。

## 12. CSS 在这个项目里的角色

React 负责“有什么”和“怎么互动”，CSS 负责“长什么样”。

极简例子：

```tsx
function Button() {
  return <button className="send-button">发送</button>;
}
```

```css
.send-button {
  background: #111;
  color: white;
  border-radius: 8px;
}
```

本项目主要样式在：

```text
react_app/src/react-entry.css
```

局部样式还有：

```text
ContextMapSidebar.polish.css
Sidebar.polish.css
SettingsView.interface.css
legacy-style.css
```

这里要注意一个维护点：

```text
react-entry.css 很大
legacy-style.css 是遗留样式
```

这说明 UI 经历过演进。以后改样式时要小心：

```text
先找组件上的 className
再去 CSS 里搜这个 className
确认是否有多个地方影响同一块 UI
```

## 14. 主聊天发送消息：从用户动作到模型回复

用户在产品里做的是：

```text
输入一句话
点击发送
看到模型逐步回复
```

技术上发生的是：

```text
Composer 输入框记录文字
        ↓
用户点击发送
        ↓
Composer 调用 onSend
        ↓
App.tsx 的 handleSend 执行
        ↓
App.tsx 确认当前会话、模型、附件
        ↓
api.ts 调用 streamMessageRequest
        ↓
后端 /api/send-message-stream 开始流式返回
        ↓
App.tsx 一边收到 delta 一边更新消息
        ↓
MessageList / MessageBubble / MessageContent 重新渲染
```

你可以把它类比成：

```text
Composer       收银台
App.tsx        店长，决定怎么处理订单
api.ts         传菜窗口
Python 后端    厨房
MessageList    上菜区域
```

这个类比不需要记死，但它能帮助你判断职责：

```text
输入框不应该自己调用模型
消息列表不应该自己保存会话
api.ts 不应该关心按钮长什么样
App.tsx 负责把这些串起来
```

## 15. 右侧上下文地图是什么

右侧上下文地图不是普通侧边栏，它是这个产品的一个核心能力。

它做的事是：

```text
把当前会话拆成节点
估算每个节点 token
用颜色/高度表示上下文重量
支持点击、拖选、minimap 定位
把选中的节点交给上下文工作区
```

相关文件：

```text
ContextMapSidebar.tsx       上下文地图主体
ContextWorkbench.tsx        上下文工作区
contextTokenWeight.ts       token 重量判断
utils.ts                    countTokens 估算 token
```

一个简单类比：

```text
普通聊天记录 = 一长串文字
上下文地图   = 把这串文字变成可管理的目录和地图
上下文工作区 = 用模型帮你整理、修改、恢复上下文
```

这部分之所以复杂，是因为它不是简单展示列表，而是在做“上下文管理产品能力”。

## 17. 看一个新需求时，应该先找哪里

如果需求是：

```text
改按钮文案、布局、显示内容
```

先找：

## 18. 当前前端架构的优点和风险

优点：

```text
主流程集中在 App.tsx，容易快速跑通
api.ts 集中管理接口，前后端通信比较清楚
types.ts 有类型约束，不是纯字符串乱传
组件已经按产品区域拆了一部分
上下文地图和工作区形成了明确业务模块
```

风险：

```text
App.tsx 太大，状态和业务逻辑过于集中
react-entry.css 太大，样式影响范围需要小心
部分中文文案看起来有编码异常，需要后续统一处理
设置页和 Provider 管理也偏大，后续可以继续拆
```

我建议后续逐步重构，而不是一次大改：

```text
第一步：从 App.tsx 拆 useChatSessions
第二步：拆 useChatStreaming
第三步：拆 useSettingsDraft
第四步：拆 useContextMapState
第五步：把大 CSS 按组件逐步收敛
```

这样做的目标不是“为了优雅而优雅”，而是让以后找 bug 更接近根源。

## 19. 你现在应该建立的最小心智模型

先别急着记所有文件。

你只要先记住这张图：

```text
用户操作
  ↓
组件触发事件
  ↓
App.tsx 处理业务逻辑
  ↓
需要后端时调用 api.ts
  ↓
Python 后端返回数据
  ↓
App.tsx 更新 state
  ↓
组件重新渲染界面
```

再对应到文件：

```text
main.tsx       启动 React
App.tsx        前端总控台
components     具体界面模块
api.ts         请求后端
types.ts       数据合同
utils.ts       通用工具
css            视觉样式
```

只要这个模型稳了，你再看复杂代码时，就不会被函数名淹没。

你可以先用三个问题读任何一个组件：

```text
1. 它在界面上是哪一块？
2. 它接收了哪些 props？
3. 用户操作后，它把事件交给谁处理？
```

读任何一个 API：

```text
1. 它请求哪个 /api 路径？
2. 它发给后端什么数据？
3. 它期望后端返回什么数据？
```

读任何一个状态：

```text
1. 这个 state 代表产品里的什么？
2. 谁会修改它？
3. 它会影响哪些组件显示？
```

这三个问题，比一开始死磕复杂函数更有效。
