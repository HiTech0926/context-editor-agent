

## 一、产品定位

**一句话描述：** 一个让你看见、理解、掌控 AI 对话上下文的开源客户端。

**核心理念：** 现有 AI 工具把上下文当黑盒，用户不知道模型实际"看到"了什么。ContextPilot 把上下文变成可视化、可操作、可管理的对象，解决"对话污染焦虑"问题。

**目标用户：**
- 重度 AI coding 工具使用者（Claude Code / Cursor / Cline 用户）
- 对上下文质量敏感、不想频繁开新对话的开发者
- 使用 API 付费、关心 token 成本的用户

---

## 二、核心问题

用户在长对话中面临的真实困境：

| 现状 | 痛点 |
|------|------|
| 对话中临时问了几个无关问题 | 主任务信号被稀释，模型注意力分散 |
| 想继续当前对话 | 但感觉上下文已经"脏了" |
| 开新对话 | 要重新交代大量背景，成本太高 |
| 用 `/compact` 压缩 | 太粗暴，怕丢失想保留的细节 |
| 想知道模型实际看到什么 | 所有客户端都是黑盒，无从得知 |

**一句话总结：** 用户需要的是精细化的上下文编辑能力，而不是"全压缩或全保留"的二元选择。

## 三、功能设计（三层递进）

### 第一层：透明化（让用户看见）

> 解决"我不知道模型看到了什么"

- **上下文可视化面板**（右侧栏）
  - 展示实际发送给模型的完整消息列表
  - 每条消息标注 token 数量
  - 用颜色区分角色：system / user / assistant / tool
  - 顶部显示总 token 数 / 窗口上限 / 使用比例进度条
  - 标注 cache 命中区间（Anthropic 的 cache breakpoint）

- **双视图模式**
  - "原始记录"：完整对话历史，永不丢失
  - "实际发送"：经过编辑/压缩后真正发给模型的内容
  - 两者之间的差异用 diff 高亮标识

### 第二层：手动操控（让用户动手）

> 解决"我想删掉那几条无关的消息"

- **消息级操作**
  - 删除：从发送上下文中移除某条消息
  - 折叠：将一条长消息压缩为摘要（用小模型生成）
  - 编辑：直接修改某条消息的内容
  - 置顶/锁定：标记为"始终保留"，压缩时不动

- **批量操作**
  - 框选一段对话，一键压缩为摘要
  - 按标签/话题筛选，批量移除某个话题的消息

- **快照与回滚**
  - 每次手动操作前自动保存快照
  - 可以回滚到任意历史版本的上下文状态

### 第三层：智能建议（让 AI 帮忙）

> 解决"我懒得自己管，但又想管好"

- **被动建议模式（默认）**
  - 当上下文超过 60% 窗口时，弹出建议卡片
  - 建议内容示例："这 5 条关于环境配置的讨论可以压缩为一句话，预计节省 1200 token"
  - 用户可以接受、修改或忽略
  - 建议由小模型生成，不消耗主模型额度

- **主动代理模式（可选开启）**
  - 每次发送前自动优化上下文
  - 用户设定规则："工具调用结果超过 500 token 自动摘要"、"超过 10 轮的历史对话自动压缩前 5 轮"
  - 所有自动操作都记录在日志中，用户可以事后查看和撤销

## 四、关键设计决策

### 代理层架构

```
用户输入
  ↓
原始对话记录（完整保存，永不修改）
  ↓
代理层（拦截 + 处理）
  ├── 手动编辑生效
  ├── 智能建议生效
  └── 自动规则生效
  ↓
实际发送内容（处理后的上下文）
  ↓
模型 API
  ↓
响应返回 → 写入原始记录
```

### 安全原则

- **原始记录神圣不可侵犯**：所有操作只影响"发送副本"
- **每步可撤销**：快照机制保证任何操作都能回退
- **默认保守**：初始状态不做任何自动处理，用户逐步开启
- **工具调用保护**：如果后续消息依赖某条工具调用结果，删除时弹出警告

### cache 友好策略

- 被编辑/删除的消息如果在 cache 命中区间内，提示用户"这条消息删除后会导致 cache 失效，预计多花 X token"
- 尽量把编辑操作集中在上下文末尾，保留头部的 cache 命中

## 五、技术栈（建议）

| 层级 | 技术 |
|------|------|
| 前端 | Next.js + Tailwind + shadcn/ui |
| 状态管理 | Zustand（双数据流：原始记录 + 发送副本） |
| Token 计算 | tiktoken（OpenAI）/ Anthropic token counting API |
| 模型接入 | OpenAI + Anthropic SDK，统一适配层 |
| 数据持久化 | 本地 SQLite 或 IndexedDB |
| 压缩用小模型 | Haiku / GPT-4o-mini（成本低、速度快） |

## 七、作品集包装策略

面试时的叙事结构：

> **发现问题** → 我是重度 AI coding 工具用户，发现所有客户端都把上下文当黑盒，长对话中的"污染"问题没有任何工具解决
>
> **分析问题** → 竞品调研（Cursor / Claude Code / Cline），发现它们要么不透明，要么只有粗粒度的 compact 功能
>
> **设计方案** → 三层递进的产品设计，从透明化到手动操控到智能建议
>
> **实现验证** → 独立完成 MVP 开发，开源发布，获得 [X] star / [X] 用户反馈
>
> **关键决策** → 为什么选择代理层架构而不是修改对话本身、为什么默认保守而不是全自动、如何平衡 cache 命中率和编辑灵活性

## 八、风险和应对

| 风险 | 应对 |
|------|------|
| 工程量超预期，AI 辅助写原创逻辑效率下降 | 严格砍功能，Phase 1 只做可视化 + 手动操控 |
| 两套数据同步容易出 bug | 数据流单向：原始记录 → 过滤规则 → 发送副本，不做双向同步 |
| 用户群体小，star 数不高 | 不影响作品集价值，重点展示产品思维而非数据 |
| 模型窗口越来越大，需求被削弱 | 窗口大了成本和注意力问题依然存在，这个需求不会消失 |
